## Active Roadmap

| ID | Failure Class | Status | Priority | Fault Boundary | Required RCA Visibility | Required Proof |
| --- | --- | --- | --- | --- | --- | --- |
| BH-001 | Same-route / no-progress inference churn | in progress | P0 | execution discipline / loop termination | RCA must show repeated route reuse, repeat count, and the no-progress termination cause rather than compressing it into generic retries | loop regression + live RCA proof |
| BH-002 | Post-success retry churn | in progress | P0 | guard / verifier precision | RCA must show that an earlier attempt succeeded, what guard or verifier intervened, whether the retry reused the same route, and when a retry was intentionally suppressed because the turn had already made substantive execution progress | guard/verifier regression + live RCA proof |
| BH-003 | Capability-truth drift for workspace / vault authoring | closed | P0 | capability truth | RCA and operator surfaces must not claim a live capability is missing when runtime/config truth says it is available, and must explain config-gated absence honestly when it is not | focused runtime tests + live authoring proof |
| BH-004 | Oversized tool surface on simple direct execution turns | closed | P0 | turn control | RCA must show why the tool surface was selected and call out when the request stayed heavier than necessary | synthesis/envelope regression + live turn proof |
| BH-005 | Under-evidenced models silently ignored | closed | P1 | routing / evidence policy | RCA must distinguish hard exclusion from insufficient capability evidence and emit recommendation-grade callouts for ignored-but-unproven models | routing tests + canonical diagnostics proof |
| BH-006 | RCA/UI loses causal atomicity | not started | P1 | RCA interpretation | RCA detail mode must preserve chronological sequence, repeat executions, retry cause, same-route reuse, fallback use, and conclusion-grade interpretation | UI contract tests + live dashboard proof |
| BH-007 | Tool execution truth drift between loop, audit table, and RCA summary | closed | P0 | capability truth / execution accounting | RCA summary and `tool_calls` audit rows must be sourced from the same execution events that actually ran, so tool-bearing turns cannot show `tool_call_count = 0` after real tool use | loop + persistence regression + live turn proof |
| BH-008 | Persistent-artifact success claims without matching artifact proof | closed | P0 | execution discipline / proof-of-work truth | RCA must show whether a claimed note/file/document creation was backed by an artifact-writing tool result, and must distinguish artifact writes from semantic-memory/authority writes | tool-policy + guard regression + live authoring proof |
| BH-009 | Replay of successful side-effecting tool calls within a turn | in progress | P0 | execution discipline / replay protection | RCA must show when a duplicate side-effecting tool call was suppressed after prior success, why it was considered replay-risky, how many replay suppressions occurred on the turn, and that the system protected the underlying resource from a second execution | loop regression + canonical RCA proof |
| BH-010 | Procedural uncertainty does not trigger applied-learning retrieval | in progress | P1 | turn control / retrieval gating | RCA must record a chronological pre-inference event showing when the framework retrieved procedural/episodic memory because it was uncertain how to carry out a task, whether it was looking for prior successes, prior failures, or both, and when uncertainty-driven retrieval was skipped | synthesis/perception regression + canonical RCA proof |
| BH-011 | Novel experiences are not captured with reuse-oriented outcome semantics | in progress | P1 | learning capture / memory distillation | RCA must record a chronological post-turn learning event showing when a turn produced reusable procedural knowledge, whether the outcome was positive, negative, or mixed, and whether that experience was merely stored or promoted for future reuse instead of remaining an unstructured one-off memory | distillation/promotion regression + canonical RCA proof |
| BH-012 | Direct execution turns degrade into read-only exploratory tool churn | in progress | P0 | execution discipline / turn control | RCA must show when the framework stopped a direct execution turn because it kept spending successful read-only context/memory/tool-inspection calls without crossing into execution progress, including the exploratory streak count, the blocked tool, and the termination cause | loop regression + canonical RCA proof |
| BH-013 | Explicit unknown session IDs fail as internal errors | closed | P0 | session authority / entry-path behavior | RCA and the request path must not continue on a phantom explicit `session_id`; the system must resolve to a durable session row or fail cleanly as not found before any message/turn persistence begins | session-resolution regression + route 404 regression + live request proof |

## Entry Template

Every new behavior-hardening item must record:

- `ID`
- `Failure class`
- `Observed turn(s)`
- `User-visible symptom`
- `Framework fault boundary`
- `Root cause`
- `Architecture rule violated`
- `Required fix`
- `Required RCA visibility`
- `Regression proof`
- `Status`

## Notes

- This roadmap is intentionally narrower than the parity roadmap.
- New entries are expected as live RCA-backed reports arrive.
- The release target is the obvious repeated failures that damage operator
  trust fastest, not a universal agent-behavior cleanup.
