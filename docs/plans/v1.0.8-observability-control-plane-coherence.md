# v1.0.8 Observability And Control-Plane Coherence Plan

## Purpose

This plan collects the post-skill-inventory inconsistencies that point to the
same architectural defect class: operator-facing status, settings, health, and
observability surfaces are reporting partial or stale control-plane truth.
These items should be addressed after the live skill inventory fix and before
release ceremony.

## Operating Rule

Every UI/status surface must answer from one of two explicit sources:

- **Runtime truth**: live daemon state, tool registry, model/provider probe,
  queue state, subagent registry, or trace/flight-recorder event stream.
- **Control-plane truth**: persisted configuration, policy, catalog metadata,
  or durable audit records.

No page may synthesize "healthy", "enabled", "available", "used", or "cost"
from static defaults when runtime evidence exists or is required.

## Ordered Work

1. **Dead-letter queue age and ownership**
   - Identify the March 15 dead-letter item and its owning subsystem.
   - Add queue-age reporting with oldest item, count, retry state, and failure
     class.
   - Add an operator action path: replay, discard with reason, or quarantine.
   - Regression: seeded dead-letter item older than threshold appears in status
     and cannot be silently ignored.

2. **Provider/model STATE health truth**
   - Stop displaying "healthy" for unreachable, unauthenticated, disabled,
     benchmark-only, or model-unavailable providers/models.
   - Separate config state from runtime health:
     - configured
     - enabled/disabled by policy
     - auth present/missing
     - endpoint reachable/unreachable
     - model usable/unusable
     - breaker open/closed
   - Regression: a provider with bad endpoint or missing key cannot render as
     healthy.

3. **Pipeline HEALTH aggregate semantics**
   - Audit why no pipeline trace has a positive HEALTH aggregate.
   - Determine whether this is real systemic degradation or a scoring/reporting
     bug.
   - Define HEALTH inputs explicitly: completion outcome, verification result,
     policy result, tool success, guard contract events, latency budget, and
     RCA class.
   - Regression: known-good synthetic trace produces positive health; known-bad
     trace does not.

4. **Flight recorder population**
   - Determine whether the flight recorder is no longer written, written to a
     different table/source, or hidden behind a UI/query mismatch.
   - Wire the ReAct/tool loop recorder to the same event source shown by the UI.
   - Regression: a turn with one tool call creates a visible flight-recorder
     entry with thought/action/observation timing.

5. **Subagent inventory disappearance**
   - Determine why zero subagents are visible despite previous registry state.
   - Distinguish "no configured subagents" from "DB query broken", "registry
     load skipped", "disabled by policy", and "UI not reading runtime roster".
   - Regression: seeded enabled subagent appears in `/status`, roster API, and
     orchestration tool output.

6. **Prompt performance cost accounting**
   - Audit prompt/cost metrics against provider usage payloads and fallback
     estimation.
   - Label estimated, provider-reported, cached, and unavailable costs
     separately.
   - Regression: missing provider token/cost data cannot render as precise cost.

7. **Settings schema guardrails**
   - Define a schema rule: fixed enumerations must render as dropdown/select
     controls, not free text.
   - Ensure every setting has title/help text, especially memory settings.
   - Add validation metadata for model policy fields such as reason codes,
     source, state, and authority.
   - Regression: schema endpoint exposes enum metadata and settings UI renders
     enum fields as selects.

8. **MCP toggle consistency**
   - Replace MCP server toggles with the same toggle component/contract used by
     the rest of settings.
   - Preserve semantic distinctions such as enabled, connected, failed, and
     reconnecting.
   - Regression: MCP toggle markup/state shape matches canonical settings
     toggle artifact.

9. **Model routing "Auto" control coherence**
   - Audit the Model Order page "Auto" toggle versus settings page
     `auto`/`route` option.
   - If redundant, remove one control and redirect users to the single owner.
   - If distinct, rename and document them so one means routing policy and the
     other means ordered fallback behavior, or whatever the actual distinction
     is.
   - Regression: changing either control has one documented persistence target
     and cannot silently disagree with the other.

10. **Delegation observability schema drift**
    - Fix `/api/observability/delegation/outcomes` and
      `/api/observability/delegation/stats` querying nonexistent columns such as
      `duration_ms`.
    - Add schema-version-aware route tests using the live migration schema.
    - Regression: delegation observability endpoints return 200 with empty
      arrays/zero aggregates on a valid empty DB instead of 500.

11. **Upgrade and mechanic repair cleanup**
    - `roboticus upgrade all` must run safe preflight and post-upgrade cleanup
      so stale updater state, stale Windows `.old*` sidecars, incomplete
      provider/skill state, and known schema drift do not leave an install
      half-upgraded.
    - `roboticus mechanic --repair` must own the same deterministic cleanup
      primitives plus DB integrity checks, orphan trace cleanup, dead-letter
      visibility checks, and stale sidecar reporting.
    - Every repair action must report one of: `repaired`, `skipped`,
      `needs_manual_action`, or `failed`.
    - v1.0.9 follow-up: expand mechanic into an intelligent repair advisor with
      ranked diagnoses, suggested commands, safe remediation bundles, and
      reversible memory lifecycle repair as defined in
      [v1.0.9 Mechanic Memory Lifecycle Repair](./v1.0.9-mechanic-memory-lifecycle-repair.md).

12. **Prompt performance tab and tuning coherence**
    - Split Prompt Performance into information tabs and a dedicated `TUNING`
      tab for controls so operators do not scroll through analysis output to
      reach actions.
    - Fix `Generate Deep Analysis` so it either produces visible analysis or
      reports the exact blocking reason.
    - Fix recommendation rendering so `Action:` cannot appear without a value.
    - Quick optimization controls must reflect current persisted/running state;
      a completed action such as enabling caching must not immediately reappear
      as the same generic recommendation unless state evidence says it is still
      disabled.

13. **Workspace agent inventory semantics**
    - Idle Agents must count only non-orchestrator agents.
    - Orchestrator agents must always be visible in the workspace inventory; if
      not actively running work, they should render as sleeping rather than idle
      pool workers.
    - The API must expose enough role/status metadata for the UI to make that
      distinction without hardcoded guesses.

14. **Immediate continuity and pending-action coherence**
    - Follow-up confirmations such as "please do" must bind to the most recent
      pending assistant-proposed action in the current session.
    - Pending actions must be represented as structured continuity state, not
      only as prose in the prior assistant message.
    - Greeting/social shortcut paths must not fire when the current user turn is
      a short confirmation/continuation of an unresolved task.
    - Regression transcript: after an assistant asks whether to proceed with
      parsing retrieved Metacritic HTML, repeated "please do" turns must resume
      that pending action instead of resetting to a greeting.

## Acceptance Criteria

- No operator-facing health/status surface reports "healthy" without matching
  runtime evidence.
- Empty observability states render as empty/zero with clear meaning, not 500s.
- Stale queues and dead observability streams are visible as defects.
- Every settings field with a finite domain is schema-backed and rendered as a
  constrained control.
- Runtime and UI inventories agree for models, providers, tools, skills,
  subagents, MCP servers, and queues.
- Upgrade and mechanic repair leave installs in an explicitly reconciled state
  or report concrete manual recovery actions.
- Prompt Performance separates tuning controls from informational output and
  every recommendation/action reflects real current state.
- Workspace agent counts distinguish sleeping orchestrators from idle
  non-orchestrator workers.
- Immediate confirmations resolve against pending action state instead of
  losing continuity and restarting the conversation.

## Suggested Test Scope

- `go test ./internal/api/routes ./internal/db ./internal/daemon ./internal/pipeline -count=1`
- Add focused route tests for every affected observability endpoint.
- Add schema/UI contract tests for settings enum metadata where available.
- Add mechanic and upgrade cleanup tests for stale sidecars, incomplete updater
  state, schema drift, and idempotent second runs.
- Add Prompt Performance route/UI contract tests for tuning tab actions,
  recommendation action labels, deep-analysis execution, and quick
  optimization state persistence.
- Add workspace inventory tests proving orchestrators are visible as
  sleeping/running and excluded from non-orchestrator idle counts.
- Add continuity regression tests for "please do"/"continue"/"go ahead"
  follow-ups after assistant-proposed next actions, including cases where a
  greeting shortcut would otherwise trigger.
- Run one live daemon smoke against observability pages after fixes land.
