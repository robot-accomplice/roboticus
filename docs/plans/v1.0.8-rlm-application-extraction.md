# v1.0.8 RLM Application Extraction

Source: "Recursive Language Models"
([arXiv:2512.24601v1](https://arxiv.org/html/2512.24601v1)).

## Purpose

This is active v1.0.8 investigation work, not feature expansion. The problem
we are trying to solve is not "add recursion," and it is not "invent a new
memory architecture." Roboticus already had the correct architecture:
working memory is active context, while long-term memory is an index/tool
surface. The regression risk is that we drift back into forced prompt injection,
generic summaries, or magic continuation scaffolds that bypass that design.

RLM is useful here as external validation of the existing Roboticus principle:
large or durable knowledge should be inspectable through tools and indexes, not
stuffed wholesale into the prompt.

The paper is useful if it helps us restore and enforce that design:

- reduce context-window pressure without replacing continuity with summaries
- preserve dense evidence that summarization would erase
- let models inspect large code, vault, memory, and trace corpora through tools
  instead of forcing all material into prompt tokens
- decide when recursive or decomposed model calls are justified by task
  complexity
- measure gains and losses through benchmark/RCA/soak evidence

## Paper Substance

RLMs treat the prompt as part of an external environment rather than forcing
the whole prompt through the model context window. The model receives metadata
about the external context, can inspect and transform that context through a
REPL, and can recursively call language models over bounded snippets.

The important result is not the Python REPL itself. The important result is
the separation of:

- large context as inspectable environment
- small model requests as focused reasoning over selected evidence
- code execution as a way to search, chunk, aggregate, verify, and preserve
  intermediate state
- recursive model calls as a selective tool for semantic transformations that
  cannot be done by code alone

The paper compares RLMs against direct long-context calls, summary agents, and
CodeAct/BM25 baselines across tasks with different context complexity:

- constant-complexity retrieval-like tasks
- multi-hop document QA
- linear dense aggregation
- pairwise/quadratic dense aggregation
- codebase understanding

The strongest evidence for Roboticus is that RLM-style approaches reduce
context rot on dense or oversized inputs, but the paper also finds that base
models can outperform RLMs on small inputs. That means RLM behavior must be a
gated mode, not a default turn loop.

## Useful RLM Concepts

| RLM concept | Roboticus interpretation |
|-------------|--------------------------|
| Prompt as environment | Treat large codebases, vaults, memory stores, traces, and benchmark artifacts as inspectable corpora rather than prompt bulk. |
| REPL/context variable | Use existing bounded tools, project inventory, search, read, and execution seams as the environment interface. |
| Context probing | Let the model inspect samples, metadata, headings, file trees, indexes, and slices before deciding how to proceed. |
| Smart chunking | Chunk by structure when possible: files, headings, sessions, traces, memory source, or semantic groups, not blind token ranges. |
| Recursive sub-calls | Use only when a subproblem requires semantic judgment over a bounded slice; prefer code for deterministic filtering/counting. |
| Variable-backed intermediate state | Persist typed intermediate findings in RCA/trace/workspace state instead of relying on hidden model scratch text. |
| Verification sub-calls | Allow only when they add new evidence; repeated self-verification without new evidence is churn. |

## Immediate Application Candidates

1. Context-as-Environment Mode

When a turn targets a large repository, vault, memory store, trace set, or
other inspectable corpus, the model should receive:

- the user task
- corpus metadata and available inspection tools
- selected samples or indexes
- strict budget for follow-up inspection
- a requirement to cite observed evidence before concluding

It should not receive the whole corpus, and it should not receive lossy summary
replacement as the first move. This is directly aligned with the Sessions >
Context footprint work: context pressure should drive elastic overhead
reduction before damaging the task.

For memory specifically, this is not new behavior. It is the original
memory-as-index/tool design restated under pressure:

- working memory is injected because it is the active conversational/action
  frame
- semantic, episodic, procedural, relationship, workflow, and fact stores are
  discoverable through the hippocampus/memory index and memory tools
- long-term memory is hydrated only when the active frame is insufficient,
  contradictory, or explicitly historical
- absence of long-term memory remains neutral unless evidence contradicts the
  current conclusion

2. Density-Aware Decomposition Gate

Recursive/decomposed calls should be allowed only when the task is dense enough
to justify them. Candidate gates:

- input exceeds usable context budget after latest user + working memory
- task requires semantic transformation over many items
- code/search can find candidates but cannot answer semantically
- prior attempt shows context rot, unsupported certainty, or missing evidence
- expected sub-call count is bounded and visible to RCA

This avoids turning every task into recursive overhead.

3. RLM-Inspired Codebase/Vault Review Soak

Create a live soak where the user starts broad and narrows over at least 25
turns:

- locate architecture docs in `~/code/roboticus`
- compare docs against source structure
- follow up with "do the next steps"
- ask for one focused subsystem comparison
- require tool-backed evidence, continuity, and no false access denial

This is the direct behavioral analogue of the paper's CodeQA/codebase
understanding use case, but evaluated through Roboticus continuity and tool
truth rather than benchmark-only prose.

4. Memory Store Inspection Without Memory Dumping

Long-term memory should remain an indexed external environment. The model
should inspect the hippocampus/memory index, recall specific entries, and
search when the active working memory is insufficient. It should not bulk-load
semantic/episodic memory into the prompt, and it should not treat absent memory
as negative evidence.

RLM reinforces the pre-existing Roboticus memory design: working memory is
active context; long-term memory is discoverable environment. Any change that
turns long-term memory back into forced prompt bulk is a regression, even if it
is justified as "RLM-inspired."

5. Recursive Evidence Distillation For RCA, Not Generic Summarization

RLM shows that summarization agents are often weaker than environment
inspection. If we distill, we should distill evidence with provenance:

- source artifact
- observed quote/path/row/trace id
- sub-question answered
- confidence and contradiction state
- whether the finding is reusable memory or only current-turn state

This is a better fit than generic "summarize until context fits."

6. Benchmark Shape Upgrade

Add benchmark families that reflect the paper's complexity distinction:

- constant lookup: one fact hidden in large irrelevant context
- linear aggregation: every entry may matter
- pairwise/relational aggregation: relationships between entries matter
- codebase understanding: answer requires several files

The scoring target should be functional correctness and evidence use, not
verbosity or phrase matching.

## Risks And Rejection Criteria

RLM also gives us several reasons to be conservative:

- Model-specific prompts matter. Reusing the same prompt across models caused
  bad trajectories in the paper. Roboticus should prefer capability profiles
  and runtime evidence over one universal recursion prompt.
- Weak coding models struggle. If a model cannot reliably reason through code
  or tool outputs, RLM mode may worsen behavior.
- Output-token pressure can break thinking models. RLM mode needs per-call
  output budgets and RCA visibility.
- Sequential recursive calls can be slow. Any implementation needs a sub-call
  budget, marginal-value tracking, and eventually async execution.
- FINAL-style sentinels are brittle. The paper's own final-answer tag
  mechanism produced strange behavior. Roboticus must use typed runtime state,
  not magic phrases.
- Verification can become churn. Repeated answer reproduction without new
  evidence is a defect, not diligence.

Reject an RLM-derived mechanism if it:

- increases false capability denials
- replaces working memory continuity with generic summaries
- hides sub-call cost or latency from RCA
- requires magic phrases to continue or finalize
- makes small/simple turns slower or less natural
- cannot be evaluated with before/after live soak evidence

## Proposed v1.0.8 Scope

The safe v1.0.8 scope is documentation, gating design, and tests/soak
definition. Runtime implementation should be limited to defects already in
flight unless the user explicitly promotes a narrow RLM-derived fix.

Accepted now:

- document Context-as-Environment Mode as a reinforcement of the existing
  memory-as-index/tool architecture
- add regression/soak requirements for dense context inspection
- require RCA to expose sub-call count, context probes, budget, and repeated
  verification churn before any RLM-like runtime mode ships
- keep RLM attribution in README, release notes, and site roadmap

Deferred until after benchmark/RCA/continuity confidence:

- recursive sub-model orchestration
- async sub-call scheduling
- operator-configurable RLM mode
- training or reinforcement from RLM trajectories

## Measurement Plan

Before any RLM-like mechanism is credited, run a before/after soak with the
same inputs and model:

- context footprint by turn
- model/tool/sub-call latency by phase
- false access/capability denial count
- continuity failures after short follow-ups
- observed-evidence coverage in final answers
- repeated verification/churn count
- memory influence classification (`-1`, `0`, `1`)
- operator-facing quality score and RCA diagnosis

Acceptance requires improvement in evidence use or continuity without a
regression in small-turn latency, false denials, or natural interaction.

## Architectural Decision

RLM is valuable for Roboticus, but the first implementation target is not
"recursive language model." It is also not a new memory injection strategy. The
first implementation target is:

```text
Context-as-Environment Mode
```

That mode should make large task context inspectable, tool-addressable,
budgeted, and RCA-visible. For memory, this means returning to and enforcing
the original design: active working memory in context, long-term memory by
index and tool. Recursive sub-calls are an optional later execution strategy
inside that mode, not the mode itself.
