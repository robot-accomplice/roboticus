# v1.0.8 Vault Capability + Liveness Intake

## Status

- Status: intake
- Date: 2026-04-24
- Source: firsthand operator transcript from the current live code path
- Scope: capability truth, sandbox path reasoning, memory-persistence intake,
  and liveness after promised execution

## Why This Belongs In v1.0.8

The 1.0.7 release materially improved verifier finalization and benchmark/RCA
truth. The next behavior-hardening pass should now catch failures where the
agent is not lying, but is still operationally wrong:

- it misunderstands which tools or paths are available
- it fails to reason over allowed subtrees
- it asks for unnecessary configuration changes
- it promises an immediate verification step and then disappears
- it treats a user correction as a generic conversational event instead of
  updating task state

These are framework-owned behavior defects. They damage operator trust even
when the final prose is not canned and not overconfident.

## Observed Transcript Shape

Good behavior worth preserving:

- The agent did not claim it could fetch blockchain logos when it believed no
  live image/web-download tooling was available.
- The agent offered a local-filesystem fallback instead of hallucinating files.
- The agent produced a useful high-level vault inventory summary after reading
  the configured vault.

Bad behavior to pipeline:

1. **Capability-boundary explanation is too coarse**
   - The agent said it lacked live web-search/image-download tools, which may
     be true for the runtime surface, but it did not expose the exact active
     tool/capability basis for that conclusion.
   - v1.0.8 should make capability denials auditable: the response and RCA
     should show whether the missing surface is web search, image search,
     download/write permissions, network policy, or provider/tool routing.

2. **Allowed-subtree reasoning failed**
   - The agent first read `/Users/jmachen/Desktop/My Vault`.
   - The operator then pointed to `Duncan/` inside that already observed vault.
   - The agent incorrectly required the subdirectory to be separately added to
     the allowlist.
   - A subpath of an allowed root is allowed unless explicitly denied by a
     narrower policy. The pipeline should not ask for new configuration when
     the requested path is a descendant of an already allowed path.

3. **Correction handling lost task semantics**
   - The operator corrected "session" to "section."
   - The agent answered with unrelated server-side config language.
   - Correction turns should update the current task model and then continue
     from the corrected semantic state, not invent a new configuration task.

4. **Memory-persistence intake stalled**
   - The operator asked the agent to examine `Duncan/` for information worth
     persisting.
   - The agent should have treated this as a memory-ingestion/evaluation task
     over an allowed subtree.
   - It instead blocked on a false allowlist assumption.

5. **Promised execution vanished**
   - The final message was "Let me test that assumption right now."
   - No result arrived for at least 34 minutes.
   - This is a liveness failure: if the agent promises an immediate test, the
     pipeline must either produce the observation, emit a bounded progress
     event, or fail with an explicit diagnostic.

## Architecture Rule Candidates

- Capability denial must be evidence-backed. The system is not allowed to say
  "I cannot access X" without a current capability/sandbox/tool-surface reason
  that RCA can inspect.
- Allowed-path checks must be subtree-aware. If a parent directory is allowed,
  descendants inherit that access unless a deny rule or write/read distinction
  says otherwise.
- User corrections must mutate task interpretation before response generation.
  A correction such as "section, not session" is task-state input, not a new
  unrelated request.
- Memory-persistence intake over an allowed subtree must enter a bounded
  inspect/evaluate/persist pipeline rather than a generic chat response.
- A promissory execution phrase such as "let me test/check/verify that" must
  bind to a liveness contract: observable tool attempt, progress heartbeat, or
  explicit failure.

## Required RCA Visibility

The RCA for this failure class should show:

- effective allowed roots and the resolved target path
- whether the requested target is a descendant of an allowed root
- active tool surface and missing capability reason, if any
- correction detected and updated task-state delta
- whether memory-persistence mode was entered
- promised execution event and either its tool result, heartbeat, timeout, or
  explicit failure

## Regression Targets

- `allowed_subdirectory_read`: configured allowed root is the vault; request
  targets `Duncan/`; expected outcome is inspection attempt, not allowlist
  refusal.
- `vault_memory_intake`: request asks to examine an allowed vault subsection
  for persist-worthy knowledge; expected outcome is bounded scan, candidate
  memory extraction, and explicit "persist / do not persist" rationale.
- `correction_semantics`: operator corrects one noun in the current task;
  expected outcome updates the active task model without inventing unrelated
  server/config requirements.
- `promised_execution_liveness`: response includes a test/check promise;
  expected outcome must include a tool attempt or bounded failure within the
  turn diagnostics window.
- `capability_denial_evidence`: missing web/image/download capability response
  must cite the exact missing runtime surface and persist that reason in RCA.

## Non-Goals

- Do not solve this with broader sandbox access.
- Do not add canned fallback prose.
- Do not bypass the pipeline with special Obsidian-only code.
- Do not persist every vault note automatically. The task is candidate
  extraction with evidence and decay/relevance policy, not wholesale ingestion.
