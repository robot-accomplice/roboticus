## P0 — Security & Data Integrity (fix before anything else)

### S1. Store.DB() Exposes Raw *sql.DB — Remove Entirely

**Problem**: `internal/db/store.go:70` exposes `func (s *Store) DB() *sql.DB` — any caller can bypass transaction discipline, skip audit logging, and ignore the query builder's safety checks.

**Fix**:
- Delete `func (s *Store) DB() *sql.DB` — no deprecation, no wrapper, gone
- Replace all callers with proper Store methods (`ExecContext`, `QueryContext`, `QueryRowContext`)
- If callers need schema introspection, add narrow `WithTx(func(*sql.Tx) error)` instead
- Add architecture test: `TestArchitecture_NoRawDBAccess` — compilation fails if DB() is re-added

**Files**: `internal/db/store.go`, all callers of `.DB()`

---

### S2. Wallet Passphrase via Environment Variable

**Problem**: `ROBOTICUS_WALLET_PASSPHRASE` at `internal/wallet/wallet.go:56` is readable by any process on the machine via `/proc/PID/environ`. This wallet moves real money on Base L2.

**Current mitigation hierarchy** (daemon/wallet_poller.go:52-85):
1. Migration-generated passphrase → keystore
2. Keystore lookup
3. Env var fallback
4. Machine-derived passphrase

**Fix**:
- Remove env var fallback and machine-derived fallback — keystore is the only source
- Passphrase resolution: keystore lookup → migration stores to keystore → fail with clear error
- If env var is set at startup: auto-migrate INTO keystore, then `os.Unsetenv()`, log warning
- Remove `walletMachinePassphrase()` — deterministic derivation from hostname is not real security
- Verify `guards_config_protection.go:19` blocks passphrase from traces/API responses
- The passphrase lives in exactly one place: the encrypted keystore. No exceptions.

**Files**: `internal/wallet/wallet.go`, `internal/daemon/wallet_poller.go`, `internal/core/keystore.go`

### S3. Delivery Queue In-Flight Recovery

**Problem**: `delivery_queue` rows in `status = 'in_flight'` are stranded on process restart — no recovery sweep. Messages silently lost.

**Schema** (db/schema.go:406-419):
```sql
status TEXT NOT NULL DEFAULT 'pending' CHECK(status IN ('pending', 'in_flight', 'delivered', 'dead_letter'))
```

**Fix**:
- Add startup recovery in daemon init: `UPDATE delivery_queue SET status = 'pending', attempts = attempts WHERE status = 'in_flight'`
- Add `next_retry_at` reset to prevent immediate retry storm
- Add dead-letter escalation after `max_attempts` exceeded
- Log recovered row count at startup
- Add integration test: insert in-flight rows, restart daemon, verify recovery

**Files**: `internal/daemon/daemon.go` (or `daemon_subsystems.go`), `internal/db/schema.go` (migration), new test

## P1 — High Impact Improvements

### H1. Pipeline.Run() — Extract Named Stages

**Problem**: `internal/pipeline/pipeline.go` Run() is ~600 lines with 13 stages inline. Stages are separated by comments but interleaved with error handling, logging, and control flow.

**Fix**:
- Extract each `// ── Stage N:` block to a named method: `(p *Pipeline) stageValidation()`, `stageInjectionDefense()`, `stageDedup()`, `stageSessionResolution()`, etc.
- Run() becomes a ~50-line orchestrator calling named stages in sequence
- Each stage method takes a `*pipelineContext` struct (session, trace recorder, config, input) and returns `(*Outcome, error)` or mutates context
- Zero behavioral change — pure refactor

**Files**: `internal/pipeline/pipeline.go` → split to `pipeline_stages.go` (already exists, extend it)

---

### H2. agentSkills Always Empty — Fabricated Telemetry

**Problem**: `pipeline.go:387-393` — `agentSkills` is declared as `[]string{}`, never populated, and suppressed with `_ = agentSkills // SA9003`. The downstream `SynthesizeTaskState()` receives empty skills, so capability-fit telemetry is fabricated zeros.

**Fix**:
- Query the skills store for agent's enabled skills at this pipeline stage
- Pass populated skills list to `SynthesizeTaskState()`
- Remove the SA9003 suppression comment
- Verify task decomposition uses skill data for capability-aware routing

**Files**: `internal/pipeline/pipeline.go`, `internal/pipeline/types_pipeline.go`

---

### H3. Token Budget len/4 Heuristic — Breaks on Non-ASCII

**Problem**: 10+ call sites use `len(content) / 4` as token approximation. This assumes 4 bytes per token (English ASCII). CJK characters are 3 bytes each but typically 1-2 tokens. Emoji are 4 bytes but 1 token. Code has high token density.

**Affected files**: `llm/compression.go:54,70`, `llm/quality.go:275`, `agent/knowledge.go:71`, `agent/compaction.go:43,82`, `agent/capability.go:75`, `pipeline/types_pipeline.go:40`

**Fix**:
- Create `internal/llm/tokencount.go` with `EstimateTokens(content string) int`
- Use `utf8.RuneCountInString()` as base (character count, not byte count)
- Apply per-script heuristic: ASCII ~4 chars/token, CJK ~1.5 chars/token, code ~3 chars/token
- Replace all 10+ call sites with `EstimateTokens()`
- Long-term: integrate tiktoken or equivalent tokenizer library

**Files**: New `internal/llm/tokencount.go`, all 10+ call sites

---

### H4. modelSupportsTools() — Static Hardcoded Blocklist

**Problem**: `internal/llm/service.go:790-802` — hardcoded string prefix list. New models require code changes and redeployment. No way for users to override.

**Fix**:
- Query model capabilities at discovery time (Ollama API returns `capabilities` field)
- Cache capability map per model
- Fall back to the hardcoded list for providers that don't report capabilities
- Add config override: `models.tool_blocklist` and `models.tool_allowlist` for user control
- Log capability source in model selection traces

**Files**: `internal/llm/service.go`, `internal/llm/provider_ollama.go`, `internal/core/config.go`

---

### H5. cmd/ — 73 Flat Files Need Subpackage Grouping

**Problem**: `cmd/` has 73 `.go` files in a single flat package. Navigation is painful, test files interleave with implementation, and the package init grows linearly.

**Fix**: Group by domain:
```
cmd/
  root.go           — cobra root command
  agent/             — agent, message, status
  models/            — models, exercise, baseline, breaker
  channels/          — channels, integrations
  config/            — config, settings, themes
  schedule/          — cron, scheduler
  wallet/            — wallet, treasury
  admin/             — daemon, health, update, parity-audit
  skills/            — skills, plugins, apps
  tui/               — tui command
```

**Constraint**: Zero behavioral change. All CLI commands keep their exact names and flags.

### H6. Treasury/Payment Pipeline — No Integration Tests

**Problem**: The treasury system (wallet polling, balance caching, transaction recording, tax payouts, swaps) has zero integration tests. The routes exist, the schema exists, but there's no test that exercises the pipeline end-to-end.

**Fix**:
- Add `internal/wallet/integration_test.go` testing: create wallet → fund → record transaction → verify balance cache → tax payout flow
- Add `internal/api/routes/treasury_integration_test.go` testing: API routes for swaps, payouts, settlements
- Use `testutil.TempStore(t)` for isolated DB per test

**Files**: New test files

## P2 — Technical Debt

### D1. Two Parallel Migration Systems

**Problem**: `internal/db/schema.go` has inline `CREATE TABLE` DDL AND `internal/db/schema_migrations.go` has numbered SQL migration files. Both run at startup. New tables are added inconsistently to one or the other.

**Fix**:
- Unify: all schema changes go through numbered migrations
- `schema.go` becomes the "version 0" baseline (initial schema only)
- Add startup validation: assert final schema matches expected structure after all migrations
- Document migration authoring in CLAUDE.md

---

### D2. Non-Configurable Context Budgets

**Problem**: L0-L3 context budget tiers are defined in config but the budget allocation logic uses hardcoded percentages in several places.

**Fix**:
- Audit all budget allocation code for hardcoded percentages
- Wire to `cfg.ContextBudget.L0` / `.L1` / `.L2` / `.L3` fields
- Add validation: percentages must sum to ≤ 1.0

---

### D3. Python Ops Scripts in Repo Root

**Problem**: 20+ Python/shell scripts (`check.py`, `health_check.py`, `monitor.py`, `disk_alert.py`, etc.) committed to repo root. These are ops tooling, not part of the Go project.

**Fix**:
- Move to `scripts/ops/` directory
- Add `.gitignore` entries for `*.json` health data files in root
- Or remove entirely if they're ad-hoc debugging artifacts (check git blame for intent)

---

### D4. coverage.out and Binaries in Git

**Problem**: `.git` is 96MB. `coverage.out` and compiled binaries may be tracked.

**Fix**:
- Add to `.gitignore`: `coverage.out`, `roboticus` (binary), `*.exe`, `*.bin`
- Run `git rm --cached` for any tracked artifacts
- Consider `git filter-branch` or BFG to clean history (separate maintenance PR)

## Theme Catalog Fixes (carried from user feedback)

### T1. Theme Install/Uninstall Buttons

**Problem**: Theme cards in Skills & Plugins catalog have no install button. Installed themes have no uninstall button. Backend `POST /api/themes/catalog/install` exists but frontend never calls it.

**Fix**:
- **Not installed**: "Install" button → `POST /api/themes/catalog/install` → swap to installed + uninstall
- **Installed**: "installed" badge + "Uninstall" button → remove from dropdown, swap to install
- **Builtin**: badge only, no uninstall (builtins can't be removed)
- If uninstalling the active theme: warn user, switch to default first
- Backend: add uninstall endpoint if missing

---

### T2. Theme Texture as Card Background

**Problem**: Theme textures shown only as "✨ body texture" in tiny 0.62rem text. The actual CSS gradient is available but not rendered.

**Fix**:
- Use the theme's `--bg` color + body texture CSS as the card background:
  ```
  background: <body-texture-value>, <theme-bg>;
  ```
- For themes with surface textures, apply to a content area within the card
- Remove the text-only "✨ body texture" hint — the visual is self-documenting
- Theme cards without textures show the solid `--bg` color (still distinctive via color preview)
