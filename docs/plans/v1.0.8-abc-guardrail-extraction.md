# v1.0.8 ABC Guardrail Extraction

Source: "Agent Behavioral Contracts: Formal Specification and Runtime
Enforcement for Reliable Autonomous AI Agents"
([arXiv:2602.22302](https://arxiv.org/pdf/2602.22302)).

## Purpose

This is active v1.0.8 investigation work, not speculative feature expansion.
The objective is to extract mechanisms that can improve or replace current
Roboticus guardrail methodology, especially where current guards are too
ad hoc, prose-shaped, or hard to diagnose through RCA.

The success bar is practical:

- improve measurable agent or memory reliability
- produce better RCA evidence than current guard names and retry booleans
- avoid canned fallback behavior entirely
- preserve the operating principle that useful imperfect answers are better
  than unnecessary no-answer outcomes

## Useful ABC Concepts

The paper defines an Agent Behavioral Contract as:

```text
C = (P, I, G, R)
```

Where:

- `P` = preconditions required before an agent action is valid
- `I` = invariants that must hold across actions or turns
- `G` = governance policies over action selection and authority boundaries
- `R` = recovery mechanisms when violations are detected

The immediately useful shift is that guards should not be only string filters
or post-hoc retry triggers. They should be runtime contracts with explicit
state, severity, recovery semantics, and evidence.

The paper also frames satisfaction probabilistically:

```text
(p, delta, k)-satisfaction
```

For Roboticus, the useful interpretation is:

- `p`: observed probability that a contract is satisfied for a model / intent /
  tool surface
- `delta`: tolerated deviation before the turn is degraded or blocked
- `k`: maximum allowed recovery window before the contract is considered failed

This maps directly to benchmark/RCA needs: a model can be recoverable but noisy,
or clean but brittle, and those should be different conclusions.

## Roboticus Mapping

| ABC concept | Roboticus target |
|-------------|------------------|
| Preconditions | Prompt/tool/action readiness checks before execution, including allowed paths, authority, required observations, and memory evidence availability |
| Hard invariants | Non-negotiable constraints such as no fabricated tool execution, no unauthorized financial action, no system prompt leakage, no invented filesystem access |
| Soft invariants | Recoverable quality issues such as repetition, unsupported certainty, weak directness, or incomplete evidence labeling |
| Governance policies | Role boundaries, orchestrator/subagent split, provider/tool eligibility, authority gates, and memory persistence policy |
| Recovery mechanisms | Scoped retry, reflection continuation, finalization from observed evidence, structured guard-exhausted error, or targeted memory correction |
| Drift metric | Guard/RCA trend over a session: increasing violations, repeated recovery, rising unsupported certainty, or memory contradiction recurrence |
| Contract composition | R-TEOR-R stages, orchestrator-to-subagent workflows, and benchmark rows where multiple inference/tool/reflection steps compose into one outcome |

## Immediate Application Candidates

1. Replace guard retry booleans with contract-shaped evidence.

Current state often reduces guard behavior to `violations`, `retry`, and
`reason`. ABC suggests emitting a richer contract event:

```text
contract_id
contract_group
severity = hard | soft
phase = retrieve | think | execute | observe | reflect | remember
precondition_state
violation_state
recovery_action
recovery_attempt
recovered
recovery_window_k
confidence_effect
```

This would make RCA more precise without changing model behavior first.

2. Separate hard failure from recoverable deviation.

Current guardrails can overreact to imperfect but useful responses. ABC gives a
clean distinction:

- hard violation: block or return structured failure
- soft violation: recover within `k` steps or degrade with evidence
- neutral/no-op: do not retry, do not punish, do not persist false learning

This aligns with the memory ternary model and the "good enough beats no
response" principle.

3. Make recovery measurable.

A guard retry should not just happen. It should answer:

- What contract failed?
- Was it hard or soft?
- What recovery path was selected?
- Did recovery occur within the allowed window?
- Did the final answer preserve useful work?

This improves benchmarking because "first attempt bad, second attempt good" is
not the same as "model is bad" or "framework is bad."

4. Introduce drift as an RCA/soak metric.

For long sessions and soaks, count contract deviation over time:

- repeated soft violations per session
- hard violations per intent class
- recovery success rate
- recovery attempts per successful turn
- unsupported-certainty trend
- false capability denial trend
- memory contradiction trend

This is more useful than aggregate pass/fail alone.

5. Use contracts to evaluate memory behavior.

Memory can be expressed as contracts:

- precondition: retrieve only when task needs continuity or non-derivable facts
- invariant: absent memory is neutral, not contradictory
- governance: derivable/mutable facts should not become durable confidence
- recovery: contradictory memory triggers uncertainty and targeted correction,
  not full wipe

This gives a better path for bad-reinforcement purge tooling.

## Explicit Non-Goals

- Do not implement a broad ContractSpec DSL before benchmark/RCA evidence is
  stable.
- Do not add model-facing canned recovery prose.
- Do not make guardrails stricter by default merely because a contract concept
  exists.
- Do not execute model-generated code without the shared sandboxed evaluator
  seam.
- Do not use ABC terminology as marketing until it has produced measurable
  reliability improvements.

## First Implementation Shape

The lowest-risk implementation path is not a new DSL. It is to reshape existing
guard/RCA evidence:

1. Add a contract-event structure behind existing guard results.
2. Classify existing guards as hard or soft.
3. Record selected recovery path and outcome.
4. Expose contract recovery metrics in RCA and benchmark reports.
5. Only then decide whether a provider/operator-editable contract schema is
   justified.

This keeps the change diagnostic-first and prevents another guardrail rewrite
that looks cleaner but fails to explain real behavior.
