# v1.0.7 Behavior Hardening Program

## Status

- Status: active
- Date: 2026-04-21
- Scope: obvious, repeatedly experienced framework-owned agent failures

## Purpose

This is not a broad “improve the agent” wishlist.

It is the release program for the obvious behavior failures we have repeatedly
experienced firsthand and can now explain with canonical RCA.

The target is not every conceivable weakness in `v1.0.7`. The target is:

1. identify when the framework is broken
2. surface why expectations were not met, whether the fault is ours or not
3. fix the obvious repeated failures that most damage operator trust
4. preserve the best extracted behavior as regression contracts

## Architectural Thesis

RCA is not only a transparency surface.

It is the intake, evidence, and verification substrate for behavior hardening.

That means:

- firsthand failures and operator-reported failures enter through canonical
  turn diagnostics
- each unique framework-owned failure class gets one roadmap entry
- fixes are not complete until RCA can explain the corrected behavior honestly
- regression tests preserve the best behavior we extract from the framework

## Operating Loop

Behavior hardening runs as a five-phase loop:

1. **Capture**
   - collect a real failed or near-failed turn
   - preserve its canonical diagnostics artifact
2. **Classify**
   - identify the primary fault boundary:
     - turn control
     - execution discipline
     - guard / verifier precision
     - capability truth
     - routing / evidence policy
     - provider / host instability
     - UI / RCA interpretation
3. **Codify**
   - create or update:
     - an architecture rule
     - one roadmap item
     - the required RCA visibility contract
     - a regression target
4. **Correct**
   - apply the smallest systemic fix at the highest responsible layer
   - upstream control-plane fixes beat downstream cleanup
5. **Verify**
   - rerun focused tests
   - recheck the live RCA
   - update release and regression artifacts

## Scope Discipline

`v1.0.7` is intentionally scoped to obvious repeated failures, not speculative
perfect behavior.

An item belongs in this program when one of the following is true:

- we experienced the failure firsthand in a live run
- an operator reported the failure and canonical diagnostics exist
- a targeted attack pass reproduced the failure with canonical diagnostics

This is the release discipline: fix what we can repeatedly demonstrate first.

## Current Workstreams

### 1. Turn Control

Stop the framework from choosing the wrong kind of turn before inference
begins.

Focus:

- task synthesis
- envelope selection
- retrieval gating
- tool-surface sizing
- applied-learning retrieval when the framework is procedurally uncertain and
  prior successful or failed workflows may already exist
- reuse-oriented capture of novel procedural outcomes so new successes and
  failures do not die as one-off turns

Both the pre-inference retrieval decision and the post-turn reuse capture must
land on the canonical RCA timeline. Operators should not have to inspect trace
metadata or memory rows to tell whether the framework consulted prior
experience, what kind of prior experience it asked for, or whether a novel
success/failure was promoted for future reuse.

### 2. Execution Discipline

Stop pointless same-route churn and require visible progress.

Focus:

- no-progress termination
- same-route retry suppression
- direct-execution suppression of repeated read-only exploration when the turn
  keeps consuming memory/context/inventory tools without crossing into actual
  execution progress
- replay protection for successful side-effecting tool calls
- direct-action decisiveness
- proof-of-work standards

Replay protection is not allowed to stay buried in loop internals. When the
framework suppresses a duplicate side-effecting tool call after prior success,
that fact must be persisted once in canonical diagnostics and projected
unchanged into operator RCA.

### 3. Guard / Verifier Precision

Make guards and verifier catch real violations, not act as late-stage janitors
for upstream mode-selection mistakes.

Focus:

- post-success retries
- repetition false positives
- unsupported-certainty handling
- action-turn vs conversational-turn distinctions

For turns that have already made substantive execution progress, guard and
verifier policy must distinguish execution-critical defects from narrative
quality defects. A completed artifact write or other successful tool-backed
action is not allowed to trigger another full inference attempt merely because
the response was stylistically repetitive or overconfident. Those concerns
still belong in RCA, but they should downgrade to recorded warnings unless the
issue means the claimed work is untrustworthy or materially incomplete.

### 4. Capability Truth

Force the UI, skill inventory, runtime tools, prompt contract, and config
preconditions to agree on what the agent can actually do.

Focus:

- capability matching
- tool registration truth
- config-gated capability honesty
- operator-visible explanations for ignored tools or models

### 5. Proof Of Work Truth

Make persistent-artifact claims depend on persistent-artifact evidence.

Focus:

- tool semantics and operation classes
- artifact-write vs authority-write separation
- proof requirements for creation/update claims
- RCA visibility when a claimed artifact was not actually created

## Initial `v1.0.7` Attack Batch

1. same-route / no-progress inference churn
2. post-success retry churn
3. capability-truth drift for workspace and vault authoring
4. oversized tool surfaces on simple direct turns
5. silent ignoring of under-evidenced models
6. false persistent-artifact success claims without matching artifact evidence

Focused authoring turns are the current active attack surface.

For note/document/file creation or update requests that are simple,
direct, and do not require authority mutation, the framework must not
inherit the generic operational always-include tool set. Those turns
need a capability-scoped tool profile built around artifact-writing
tools first, only the minimum runtime context needed to execute them,
and retrieval only when continuity evidence demands it.

## Closure Standard

A behavior issue is only closed when all of the following are true:

- architecture is updated
- documentation is updated
- the framework behavior is corrected
- RCA explains the corrected behavior honestly
- regression tests preserve the best observed behavior

That is how transparent RCA becomes actionable RCA and, later, automated
recommendation and self-healing behavior.
