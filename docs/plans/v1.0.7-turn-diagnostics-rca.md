# v1.0.7 Turn Diagnostics RCA

## Status

- Status: proposed
- Date: 2026-04-19
- Scope: turn-level triage / forensics / operator truth surfaces

## Context And Drivers

The primary product need for `v1.0.7` is not "more logs." It is a single,
systemic turn-diagnostics mechanism that can answer, for any bad turn:

- what the system tried
- what actually happened
- where time went
- why fallback/retry/tool/guard behavior happened
- what the user should be shown about that behavior

This mechanism must serve three consumers without forking runtime truth:

1. operator / engineering forensics
2. chat-channel targeted triage feedback
3. WebUI layman-friendly flow visualization
4. actionable remediation recommendations the user can apply to reduce recurrence

If those three consumers are fed by different reconstruction paths, the system
will drift again and observability will become an approximation instead of a
runtime truth surface.

## Target Scenario

The concrete RCA target for this work is the Duncan interaction pattern
observed on 2026-04-19:

- command-shaped turns (`/breaker`, `/clear`, `/help`) are followed by clearly
  incoherent or stale-context responses
- the agent appears unable to distinguish command intent from ordinary
  conversation
- filesystem/runtime-limit narration repeats without progressing the task
- the operator cannot confidently tell whether the failure was caused by:
  - inference structure
  - context overload / bad context composition
  - routing/model choice
  - tool-usage instability
  - resource constraints on the active model

This interaction class is the acceptance target for the new diagnostics
system.

The new RCA is successful only if, after implementation, Roboticus can:

1. explain why that interaction degraded
2. distinguish among the competing plausible causes with evidence
3. surface that explanation to the operator and user appropriately
4. emit concrete recurrence-reduction recommendations that can be applied from
   the WebUI

## Incident Evidence

The 2026-04-19 Duncan logs already establish that this class of failure is
system-owned and multi-factor:

- local Ollama attempts timed out repeatedly with `context deadline exceeded`
- local Ollama also returned `model runner has unexpectedly stopped`, which is
  explicitly suggestive of resource instability
- the same degraded turn carried request shapes near the effective ceiling,
  including `tokens_in=8192`, large message counts, and full tool rosters
- guard and verifier retries then re-issued already unhealthy turns instead of
  simplifying them
- later turns mixed clear tool-appropriate requests with repeated sandbox
  narration and malformed placeholder content such as `[assistant message]`

That means the target diagnostic system must attribute at least four interacting
dimensions on a bad turn:

1. model/provider attempt outcomes
2. request/context pressure
3. retry/fallback amplification
4. tool-selection / tool-execution failure mode

Anything less will recreate the same ambiguity that blocked diagnosis in the
Duncan incident.

## Test Strategy

The Duncan chat transcript plus the corresponding logs should be treated as the
first regression sketch for this work.

The test goal is not to reproduce the entire session verbatim. The goal is to
capture its causal shape in executable form:

- degraded local-model attempts
- explicit fallback into another model/provider
- retry pressure from guards and/or verifier
- a single persisted turn-diagnostics artifact with evidence-bearing events
- stable summary fields that can feed both operator and user-facing projections

This lets the RCA stay tied to a real observed failure class instead of
drifting into generic observability work.

## Current State

The current runtime already emits useful telemetry, but it is fragmented.

### Existing strengths

- `TraceRecorder` gives the pipeline a structured span surface with per-stage
  metadata and total duration. See [internal/pipeline/trace.go](/Users/jmachen/code/roboticus/internal/pipeline/trace.go:27).
- `storeTraceWithArtifacts(...)` persists stage spans plus optional
  `react_trace_json` and `inference_params_json` to `pipeline_traces`. See
  [internal/pipeline/pipeline.go](/Users/jmachen/code/roboticus/internal/pipeline/pipeline.go:563).
- The schema already preserves adjacent telemetry surfaces:
  `pipeline_traces`, `tool_calls`, `model_selection_events`,
  `context_snapshots`, and `turn_feedback`. See
  [internal/db/schema.go](/Users/jmachen/code/roboticus/internal/db/schema.go:695) and
  [internal/db/schema.go](/Users/jmachen/code/roboticus/internal/db/schema.go:572).
- WebSocket snapshots reuse HTTP handlers rather than inventing another data
  path. See [internal/api/ws_topics.go](/Users/jmachen/code/roboticus/internal/api/ws_topics.go:11).

### Architectural break

There is no single authoritative "turn explanation" artifact.

Instead, the turn story is scattered across:

- `pipeline_traces.stages_json`
- `pipeline_traces.react_trace_json`
- `pipeline_traces.inference_params_json`
- `tool_calls`
- `model_selection_events`
- `context_snapshots`
- ad hoc log lines in the in-memory log ring buffer

The UI and API then reconstruct different partial views from those pieces.

This means current observability is not decision-grade even for operator
forensics in practice. It can support hypotheses; it cannot reliably support
root-cause conclusions strong enough to justify changing development strategy,
runtime policy, or model inventory.

## Findings

### [P0] Current telemetry is insufficient to distinguish runtime-ownership failures from model-limit failures

Concrete example:

- A turn may degrade because inference was structured poorly.
- Or because the selected model was overloaded with too much context.
- Or because retrieval/context assembly sent the wrong information mix.
- Or because tool-eligibility/tool-execution framing broke the model's ability
  to respond coherently.
- Or because resource pressure on the host changed latency/quality enough to
  destabilize the turn.

Current telemetry does not let us confidently separate those cases.

That is a release-blocking observability defect, because the system cannot take
responsibility for its own failures if it cannot tell whether the problem was:

- request structure
- context volume / context shape
- routing choice
- model suitability under current machine constraints
- fallback / retry behavior
- tool orchestration behavior

This is exactly the gap surfaced by the Duncan example:

- clearly wrong conversational carryover after `/breaker`
- incoherent answer selection relative to the command semantics
- repeated runtime-limit narration without forward progress
- apparent confusion around tool / filesystem / subagent handling

Those symptoms could be caused by multiple different runtime seams. Today the
system does not emit enough authoritative evidence to discriminate among them.

Therefore:

- "model quality concern" is not an acceptable terminal explanation
- Roboticus must own the diagnosis boundary
- `v1.0.7` must make these cases attributable with evidence strong enough to
  drive product and runtime changes

### [P1] The canonical turn artifact is fragmented across unrelated tables and routes

Evidence:

- `pipeline_traces` stores stage spans plus optional JSON sidecars, but tools,
  model selection, context, and feedback live elsewhere. See
  [internal/db/schema.go](/Users/jmachen/code/roboticus/internal/db/schema.go:572),
  [internal/db/schema.go](/Users/jmachen/code/roboticus/internal/db/schema.go:586),
  [internal/db/schema.go](/Users/jmachen/code/roboticus/internal/db/schema.go:695).
- The "trace detail" route returns only parsed stages, not the full stored
  artifact. See [internal/api/routes/traces.go](/Users/jmachen/code/roboticus/internal/api/routes/traces.go:162).
- Turn-level detail is assembled through separate endpoints for tools, context,
  and model selection. See
  [internal/api/routes/turn_detail.go](/Users/jmachen/code/roboticus/internal/api/routes/turn_detail.go:15),
  [internal/api/routes/turn_detail.go](/Users/jmachen/code/roboticus/internal/api/routes/turn_detail.go:80),
  [internal/api/routes/turn_detail.go](/Users/jmachen/code/roboticus/internal/api/routes/turn_detail.go:161).

Impact:

- There is no one object that can be declared authoritative for "what happened
  on this turn."
- Any chat-facing triage narrative or layman flow must reconstruct causality
  after the fact from multiple surfaces.
- Reconstruction rules will drift from runtime truth.

### [P1] The trace model is stage-timing oriented, not causality oriented

Evidence:

- `PipelineTrace` only contains ordered spans with `name`, `duration_ms`,
  `outcome`, and freeform metadata. It has no event IDs, no parent/child edges,
  no absolute start offsets, and no typed causal relationship model. See
  [internal/pipeline/trace.go](/Users/jmachen/code/roboticus/internal/pipeline/trace.go:27).
- `ReactTrace` is stored as another sidecar structure rather than part of the
  same typed event stream. See
  [internal/pipeline/pipeline.go](/Users/jmachen/code/roboticus/internal/pipeline/pipeline.go:571).

Impact:

- A waterfall is possible.
- A trustworthy layman flow diagram is not.
- Retry/fallback/tool/guard causality must be inferred from ordering and string
  metadata instead of represented directly.

### [P1] The UI/WebSocket path consumes summary routes, not a single turn-diagnostics artifact

Evidence:

- The `traces` WebSocket topic snapshots `routes.ListTraces(...)`, which is the
  summary list route, not a canonical turn-explanation payload. See
  [internal/api/ws_topics.go](/Users/jmachen/code/roboticus/internal/api/ws_topics.go:16).
- `/api/observability/traces` and `/api/observability/traces/{id}/waterfall`
  also expose only stage lists. See
  [internal/api/routes/observability.go](/Users/jmachen/code/roboticus/internal/api/routes/observability.go:12).

Impact:

- Realtime surfaces can stay transport-consistent while still being
  explanation-incomplete.
- The UI is forced to fetch and compose multiple partial artifacts instead of
  subscribing to one authoritative turn-diagnostics object.

### [P1] The log ring buffer is not strong enough to support real forensics or product strategy

Evidence:

- `LogRingBuffer` stores freeform log entries with arbitrary fields, but there
  is no enforced turn/session correlation contract and no persistence into a
  turn-owned artifact. See [internal/api/logbuffer.go](/Users/jmachen/code/roboticus/internal/api/logbuffer.go:10).

Impact:

- Logs can expose interesting symptoms during a live incident.
- Logs still cannot support confident attribution of failure cause.
- Logs cannot serve as the authoritative substrate for chat-safe triage, a
  deterministic UI flow, or product-level recommendation generation.
- Important causality can still live only in transient text or not be emitted
  at all.

### [P2] Even the trace routes are lossy relative to what the database already stores

Evidence:

- `pipeline_traces` stores `react_trace_json` and `inference_params_json`, but
  `GetTrace`, `TraceWaterfall`, and `ExportTrace` only return stage lists plus a
  few envelope fields. See
  [internal/db/schema.go](/Users/jmachen/code/roboticus/internal/db/schema.go:695),
  [internal/api/routes/traces.go](/Users/jmachen/code/roboticus/internal/api/routes/traces.go:162),
  [internal/api/routes/observability.go](/Users/jmachen/code/roboticus/internal/api/routes/observability.go:60).

Impact:

- The system already discards part of its own persisted truth at the API
  boundary.
- Any future channel/UI explanation built on those routes starts from an
  incomplete artifact.

## Root Cause

Observability has been treated as a family of useful adjacent artifacts rather
than a single turn-owned explanation model.

That produces good local improvements:

- richer spans
- stored inference params
- stored model selection events
- tool call records
- log capture

But it still leaves the most important invariant unowned:

`A turn must produce one authoritative diagnostics artifact from which every
other triage/forensics surface is derived.`

Without that invariant, every new observability surface becomes another
reconstruction path.

A second root cause sits underneath that one:

`The system does not yet capture the evidence needed to decide whether a bad
turn was caused by runtime structure, context overload, routing choice, model
unsuitability, or machine/resource constraints.`

So the problem is not merely fragmented telemetry. It is also missing telemetry.

## Decision

`v1.0.7` should introduce a single authoritative **Turn Diagnostics Artifact**
(TDA) backed by a typed turn event stream.

Logs, chat-facing triage, and WebUI flow views must all derive from that same
artifact.

This artifact must be decision-grade:

- strong enough to support RCA
- strong enough to support development-strategy changes
- strong enough to support model-pool/routing changes
- strong enough to emit evidence-backed user recommendations

### TDA requirements

Each turn must persist:

- stable `turn_id`, `session_id`, `channel`
- overall outcome and user-visible result classification
- an ordered typed event stream with timestamps / offsets
- explicit causal links for retry/fallback/tool/guard relationships
- actor/owner for each event (`pipeline`, `router`, `provider`, `tool`,
  `guard`, `verifier`, `channel`)
- operator detail payload
- user-safe summary payload
- recommendation inputs sufficient to derive recurrence-reduction guidance

### Minimum event types

- `turn_started`
- `context_built`
- `routing_decided`
- `model_attempt_started`
- `model_attempt_finished`
- `fallback_triggered`
- `context_pressure_assessed`
- `tool_eligibility_decided`
- `tool_call_started`
- `tool_call_finished`
- `guard_failed`
- `guard_retry_scheduled`
- `verifier_finished`
- `resource_pressure_observed`
- `recommendation_emitted`
- `response_finalized`
- `turn_finished`

### Required event fields

- `event_id`
- `turn_id`
- `seq`
- `type`
- `at_ms`
- `duration_ms` where applicable
- `parent_event_id` where applicable
- `status`
- `operator_summary`
- `user_summary`
- `details_json`

### Recommendation model

The canonical artifact must support a second-order output: not just "what went
wrong," but "what should change so this is less likely to happen again."

Recommendations must be:

- evidence-backed by the same stored event stream
- scoped to a concrete failure mode or resource constraint
- actionable through the product surface, not just prose
- explainable in both operator language and layman language

Recommendation categories should include at least:

- model routing changes
- provider/model disable or deprioritize actions
- provider/model add or enable suggestions
- timeout / retry / fallback policy changes
- context-budget / memory-pressure mitigations
- tool / MCP / plugin health interventions
- machine-capacity or deployment-environment warnings

Examples:

- "This turn spent 84% of latency in repeated cold starts on model X; recommend
  deprioritizing X on this machine."
- "Fallback consistently escapes to provider Y because provider X times out
  under current memory pressure; recommend disabling X for this host profile."
- "Large-context tasks exceed available RAM on the local model set; recommend
  adding a remote model or shrinking the candidate pool for L3 turns."
- "Tool-call heavy turns fail after model Z emits malformed tool invocations at
  materially higher rate than peers; recommend removing Z from tool-eligible
  routing."

### Missing telemetry that v1.0.7 must add

The current system does not emit enough evidence to answer the real questions.
The canonical artifact must add at least:

- per-attempt inference timing
  - queue delay
  - provider latency
  - cold-start / warm-start classification
  - timeout / cancellation cause
- context-shape telemetry
  - token counts by source
  - retrieval contribution by tier
  - tool roster size
  - prompt/history compaction facts
  - whether the request was likely above the model's usable context quality
    threshold even if technically within token budget
- resource telemetry at attempt time
  - host memory pressure
  - CPU/load pressure
  - concurrency / in-flight model work
  - model residency / readiness state for local models
- causal retry/fallback telemetry
  - why a retry happened
  - why fallback happened
  - what condition made the prior attempt unacceptable
- tool-usage stability telemetry
  - whether the model was offered tools
  - whether it attempted tool use coherently
  - whether parsing/tool-call formatting failed
- final degradation attribution
  - stable diagnosis codes with confidence, not just prose

## Proposed Architecture

### 1. Runtime ownership

Add a pipeline-owned `TurnDiagnosticsRecorder` that is the only place allowed
to record turn diagnostics.

Existing trace/span helpers can remain, but they become inputs to the recorder,
not the final public artifact.

That recorder must be fed not only by pipeline stages, but also by the actual
inference/routing/tool/runtime ownership points where failure causality is
decided.

### 2. Persistence ownership

Persist diagnostics as one turn-owned surface, not a fan-out of unrelated route
joins.

Recommended shape:

- `turn_diagnostics`
  - one row per turn
  - summary / status / top-level timings / narrative fields
- `turn_diagnostic_events`
  - ordered typed events per turn
  - causal edges via `parent_event_id`

The current `pipeline_traces`, `tool_calls`, and `model_selection_events` can
remain for compatibility and targeted queries, but they should be treated as
supporting projections, not the canonical explanation artifact.

### 3. Presentation ownership

Derived surfaces only:

- Operator logs: projection of TDA events into zerolog lines
- Chat triage feedback: projection of the same events into concise
  user-appropriate explanations
- WebUI flow: projection of the same events into a layman-friendly node/edge
  graph plus expandable operator detail
- WebUI remediation actions: projection of the same events into explicit,
  user-triggerable recommendations with safe apply paths

The UI must not have to invent causality by joining trace stages, tools, and
model selection records on its own.

### 4. Recommendation ownership

Recommendation generation must not live only in the UI.

The pipeline/runtime owns the evidence artifact. A recommendation engine should
consume that artifact and emit structured remediations, for example:

- `kind`: `disable_model`, `deprioritize_model`, `add_model_candidate`,
  `adjust_timeout`, `reduce_parallelism`, `pin_provider`, `trim_context_policy`
- `subject`: model / provider / route / tool / host profile
- `reason_code`: stable machine-readable diagnosis
- `confidence`
- `evidence_event_ids`
- `operator_explanation`
- `user_explanation`
- `apply_payload`

That keeps recommendation logic systemic and lets the WebUI render "Apply"
controls without inventing its own diagnosis layer.

## First Implementation Slice

### Slice A: make the runtime produce an authoritative event stream

- introduce `TurnDiagnosticsRecorder`
- emit typed events alongside current spans
- capture routing decision, each model attempt, fallback trigger, context
  pressure, tool eligibility, tool call, guard retry, verifier result, resource
  pressure, and final response classification

### Slice B: persist and serve one canonical turn diagnostics route

- add `turn_diagnostics` and `turn_diagnostic_events`
- add `turn_diagnostic_recommendations`
- add `/api/turns/{id}/diagnostics`
- return:
  - summary
  - event stream
  - layman flow nodes/edges
  - operator detail
  - structured recommendations with `apply_payload`

### Slice C: rebase existing surfaces onto that route

- WebSocket traces/topic should publish the canonical diagnostics summary for
  changed turns
- observability page and waterfall should be projections of TDA
- chat-channel targeted triage should be generated from TDA
- recommendation cards / apply-actions in the WebUI should be generated from
  TDA-backed recommendations

## Trade-Offs

### Gains

- one truth surface for all turn triage consumers
- real causal attribution instead of string-metadata inference
- easy expansion into user-facing explanations without re-instrumenting
- less route drift and less dashboard-specific logic
- a real closed loop from incident evidence to recurrence-reduction action
- evidence strong enough to decide whether Roboticus should change inference
  structure, context policy, routing policy, or model inventory

### Costs

- new schema and migration work
- recorder wiring across pipeline, router, tool execution, and guards
- temporary duplication while old trace routes remain for compatibility

### Risks

- overfitting the first event taxonomy and having to evolve it
- leaking operator-only detail into user-facing payloads if safety boundaries
  are not explicit
- trying to boil the ocean by replacing every old surface at once
- emitting aggressive recommendations that are not safe to auto-apply without
  confidence thresholds and explicit user confirmation

## Non-Goals

- replacing every existing trace route immediately
- deleting `pipeline_traces` in `v1.0.7`
- making raw logs the source of truth

## Acceptance Criteria

The RCA is only considered resolved when all of the following are true:

1. A single route can answer "why did this turn behave this way?" without the
   caller joining other observability endpoints.
2. That route contains enough evidence to discriminate between:
   - inference-structure failure
   - context-overload / context-shape failure
   - routing/model-selection failure
   - model unsuitability under current machine constraints
   - tool-orchestration failure
3. The same persisted artifact can render:
   - operator detail
   - chat-safe triage summary
   - layman flow diagram
   - structured recurrence-reduction recommendations
4. Fallback/retry/tool/guard causality is explicit in stored events, not
   inferred from string search over JSON blobs.
5. Existing trace pages are projections of the canonical artifact rather than
   parallel truth surfaces.
6. At least one recommendation class is user-applicable in the WebUI from the
   same canonical artifact, without bespoke dashboard-only diagnosis logic.

## Recommendation

Do not continue adding targeted logging directly to route handlers, dashboard
views, or channel adapters.

The next implementation work should be:

1. define the `TurnDiagnosticsRecorder` interface and event taxonomy
2. add the canonical diagnostics tables
3. implement `/api/turns/{id}/diagnostics`
4. migrate the dashboard traces/flow and channel triage to consume that
   artifact
