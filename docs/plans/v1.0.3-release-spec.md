# v1.0.3 Release Specification

**Date**: 2026-04-12
**Status**: Planning
**Thesis**: Fix what's broken, clean what's bloated, ship what's new. Core feature development deferred to v1.0.4.

## Release Tracks

### Track 1: Bug Fixes & Parity (7 items)

#### I1. Episodic Memory Rendering — Aborted JSON

**Problem**: Tool outputs (JSON) truncated to 150 chars via `safeUTF8Truncate()` at `internal/agent/memory/manager.go:122`. Stored as-is, returned raw in API (`memory.go:95`) and injected into LLM context (`retrieval.go:298`).

**Fix**:
- Detect JSON content before truncation
- For JSON: extract a natural-language summary (tool name + key fields) instead of raw truncation
- For non-JSON: keep current truncation behavior
- Add rendering sanitization in the API response layer

**Files**: `internal/agent/memory/manager.go`, `internal/api/routes/memory.go`, `internal/agent/memory/retrieval.go`

---

#### I2. Skills Catalog — Missing Plugins & Themes Sections

**Problem**: `GetSkillsCatalog()` at `internal/api/routes/session_detail.go:241` returns empty plugins array and only builtin themes.

**Fix**:
- Populate plugins section from plugin registry inline (not deferred to dashboard)
- Merge catalog themes (with install status) into the response
- Single unified catalog endpoint response

**Files**: `internal/api/routes/session_detail.go`

---

#### I3. Agent Roster — Gaps vs Rust

**Problem**: Go roster (`workspace.go:46-83`) returns only name/model/enabled/state/activity/role. Rust returns voice parameters, missions, enabled skills by kind, directives, personality metadata.

**Fix**:
- Load OS/firmware/directives from workspace path
- Enrich roster entries with voice config, missions, skill assignments
- Match Rust's `workspace_roster.rs` response shape

**Files**: `internal/api/routes/workspace.go`, possibly new `internal/api/routes/roster.go` if workspace.go is too large

---

#### I4. Cron Scheduled Reports — Not Delivered

**Problem**: Pipeline outcome discarded at `daemon.go:1018` (`_, err :=`). Delivery fields exist in schema but are orphaned — never read, never set, never routed.

**Fix** (6 integration points):
1. Add `DeliveryMode`/`DeliveryChannel` fields to `CronJob` struct (`schedule/scheduler.go`)
2. Include delivery fields in worker's SELECT query (`schedule/worker.go:112`)
3. Support delivery fields in cron creation/update APIs (`routes/cron.go`, `tools/cron.go`)
4. Capture pipeline outcome in daemon executor (`daemon.go:1018`)
5. Route outcomes through delivery system based on job config
6. Default `delivery_mode` for existing jobs to the channel that created them

**Files**: `internal/schedule/scheduler.go`, `internal/schedule/worker.go`, `internal/daemon/daemon.go`, `internal/api/routes/cron.go`, `internal/agent/tools/cron.go`

---

#### I5. Pipeline Trace / Flow — Richness Gap

**Problem**: Not all pipeline stages are instrumented with annotation calls. Flow format missing guard decisions, model selection rationale, memory retrieval stats, policy evaluations.

**Fix**:
- Audit each pipeline stage for `AnnotateTrace()` calls
- Add annotations for: guard outcomes (pass/fail/reason), policy engine decisions, memory retrieval strategy + hit counts, routing selection rationale, context budget allocation
- Verify flow format includes all metadata in the response

**Files**: `internal/pipeline/pipeline_stages.go`, `internal/pipeline/guards_*.go`, `internal/pipeline/trace.go`, `internal/api/routes/traces.go`

---

#### I6. Flight Recorder — Empty Sessions Shown

**Problem**: `ListSessions()` at `sessions.go:14-47` returns ALL sessions unconditionally. No filtering for sessions with associated flight records.

**Fix**:
- Add a flight-recorder-specific query/filter: `WHERE EXISTS (SELECT 1 FROM react_traces rt JOIN pipeline_traces pt ON pt.id = rt.pipeline_trace_id WHERE pt.session_id = s.id)`
- Either new endpoint or query parameter on existing listing

**Files**: `internal/db/route_queries.go`, `internal/api/routes/sessions.go`

---

#### I7. User Weight Configuration — E2E Verification

**Problem**: Full path appears intact but needs integration-level verification from running instance. Weight changes should be visible in pipeline traces.

**Fix**:
1. Add weight provenance to pipeline traces (which weights were active when model was selected)
2. Add integration test: set weights → trigger inference → verify trace shows weight-influenced selection
3. Verify persistence survives restart in smoke test
4. Add weight snapshot to `/api/routing/profile` response (current effective vs persisted)

**Files**: `internal/pipeline/trace.go`, `internal/llm/router.go`, `smoke_test.go` or new integration test

### Track 2: Clean Code / Structural Debt (5 items)

#### D1. daemon.go Split (1600 → ~3-4 files)

Split `internal/daemon/daemon.go` by responsibility:
- `daemon.go` — lifecycle (Start/Stop/Run, field declarations) ~400 lines
- `daemon_adapters.go` — executor/stream/nickname adapters ~400 lines
- `daemon_subsystems.go` — background workers (cron, treasury, delivery, health) ~400 lines
- `daemon_init.go` — initialization, config loading, subsystem wiring ~400 lines

**Constraint**: Zero behavioral change. All tests must pass unchanged.

#### D2. Top File-Size Offenders (5-6 files)

Priority splits (worst offenders first):
- `db/route_queries.go` (1142) → split by domain (session, memory, stats)
- `core/config.go` (1120) → extract validation, subsystem defaults
- `db/schema.go` (1008) → extract migration groups
- `llm/service.go` (871) → extract stream wrappers, cache layer
- `api/routes/admin.go` (640) → extract webhook handlers

#### D3. OAuth Goroutine Context Leak

Fix `internal/core/oauth.go:148` — wrap `server.Serve()` with context monitoring so server shuts down when context cancels.

#### D4. Nickname Refinement Off-Pipeline

Either move nickname refinement through pipeline as a post-turn stage, or document it as an approved off-pipeline exception with rationale comment.

#### D8. Preset Doc Comments (Gap 7)

Add rationale doc comments to each preset function in pipeline config explaining why stages are disabled/enabled per channel type.

### Track 3: New Capabilities

#### Apps (2)

- Agent personality grounded in Eastern philosophical traditions
- FIRMWARE with contemplative voice, Socratic questioning, mindfulness-aware responses
- Themes: Buddhist, Taoist, Hindu, Confucian perspectives

**Western Philosophy App** (`roboticus-apps/western-philosophy/`)

- App manifest with appropriate model requirements

Both apps follow the Tabletop GM pattern: separate DB per profile, TOML manifest, OS/FIRMWARE/DIRECTIVES structure.

#### Plugins (2)

**Codex CLI Plugin** (roadmap P.13)
- Typed wrapper around external AI CLI tools (Claude Code, Codex, etc.)
- Explicit parameters, timeout/output limits, policy classification
- Structured result parsing with provenance
- Registered via ScriptPlugin pattern with `ExecuteTool`

**Web Research Plugin**
- Structured web search with configurable search backends
- Page content extraction and summarization
- Citation tracking with source attribution
- Rate limiting and caching
- Policy-governed with appropriate risk classification

#### MCP Client Integrations

The agent should be able to connect to and use external MCP servers as a **client**. This is the inverse of the existing `McpServerRegistry` (which exposes our tools to others).

**Architecture**:
- MCP client connection manager in `internal/mcp/client.go`
- Tool discovery: query connected MCP servers for available tools
- Tool routing: MCP tools registered in the agent's tool registry with `source: "mcp"`
- Policy integration: MCP tool calls governed by the same policy engine
- Configuration: MCP server connections defined in config (`mcp_servers` section)

**Initial integrations to validate the pattern** (configured, not bundled):
- Playwright (browser automation)
- File system operations
- External API connectors

#### Themes (3)

**ThemeSolarized** — Solarized dark palette
- Base03 background, Base0 body text, Cyan accent, Yellow highlights
- Popular developer aesthetic, excellent readability

**ThemeCyberpunk** — Neon aesthetic
- Dark background, hot pink accent, cyan secondary, green terminal text
- Differentiator, fits AI/agent aesthetic

**ThemeMinimal** — Accessibility-focused
- Grayscale palette, no icons/nerdfonts, bold-only emphasis
- Maximum terminal compatibility, high contrast

**Files**: `internal/core/style.go` — extend `ThemeVariant` enum, add color/icon/glyph methods
