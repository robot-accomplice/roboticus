# v1.0.9 Mechanic Memory Lifecycle Repair

## Problem

Memory repair needs to remove likely-garbage records from active operation
without destroying forensic evidence. The immediate category includes stale,
untrustworthy, orphaned, superseded, incorrectly reinforced, or derived memory
records that no longer serve a useful runtime, diagnostic, or historical
purpose.

The current mechanic can perform deterministic v1.0.8 cleanup, but it does not
yet explain why questionable memories survived lifecycle pressure, nor does it
offer a reversible review path before final deletion.

## Business Requirement

Mechanic must provide reversible, explainable memory hygiene:

- identify memories or derived memory records that should no longer influence
  prompts, retrieval, scoring, or operator-facing conclusions
- explain why those records remained active or prompt-visible
- remove unfit records from runtime influence without immediate irreversible
  deletion
- preserve enough original row data for restore, audit, diff, and lifecycle
  forensics
- permanently delete quarantined records only after a user-configurable
  retention period

## Required Semantics

Mechanic should distinguish these outcomes:

- `active`: record still serves current runtime, diagnostic, or historical
  purpose
- `disabled_in_place`: authoritative row remains in its source table but no
  longer participates in runtime use
- `quarantined`: row was copied into mechanic quarantine and removed from active
  query surfaces
- `rebuildable_deleted`: derived row was deleted because it can be rebuilt and
  has no independent forensic value
- `needs_manual_action`: mechanic cannot safely decide without operator review
- `expired`: quarantined row passed retention and was permanently deleted

## Mechanic Quarantine

Introduce a mechanic-owned quarantine surface for reversible removal. The
working table shape should capture:

- quarantine id
- timestamp of removal
- expiration timestamp
- source table name
- source row id when available
- reason and lifecycle diagnosis
- repair run id
- complete row payload as JSON
- row hash for tamper/equality checks
- restoration timestamp and result if restored

The operator-facing name may be "quarantine", "recovery bin", or similar, but
the product meaning is not limited to suspected malicious data. It means a row
has been removed from active operation pending review, restoration, or expiry.

## Memory Lifecycle Explainability

For any questionable memory, mechanic should be able to report:

- memory tier and source table
- current lifecycle state and state reason
- confidence, importance, and recency signals
- last reinforced or last retrieved evidence when available
- retrieval/index/FTS visibility
- promotion, supersession, or deduplication links when available
- decay/prune rule that should have handled the record
- reason mechanic recommends active, disabled, quarantined, or manual review

## Constraints

- No full memory wipe unless targeted repair is proven insufficient.
- No silent mutation. Every mechanic action must report `repaired`, `skipped`,
  `needs_manual_action`, or `failed`.
- Runtime prompt-facing derived rows must not remain active merely because they
  were copied to quarantine.
- Configurable retention is required before permanent deletion.
- Restore must validate that the target schema still matches the quarantined
  payload or report manual action.

## v1.0.8 Boundary

v1.0.8 remains limited to deterministic repair primitives that stop known
runtime poisoning, including stale derived memory-index and FTS rows. v1.0.9 is
where mechanic becomes lifecycle-aware, reversible, and diagnostic enough to
explain why memory hygiene failed upstream.
