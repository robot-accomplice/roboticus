# v1.0.7 Benchmark/RCA Hardening Plan

## Release Thesis

- `v1.0.7` is primarily a **benchmark truth + RCA truth** release.
- The first priority is not ranking more models. The first priority is making
  sure our benchmark, verifier, and soak surfaces can distinguish:
  - framework-owned failure
  - harness-owned failure
  - benchmark-design failure
  - provider/protocol failure
  - genuine model capability failure
- Until that distinction is trustworthy, model-efficacy output is partially a
  measurement of Roboticus itself, not just the model under test.

## Priority Order

1. **Benchmarking + RCA as one coupled program**
   The benchmark is currently measuring us as much as the model. These tracks
   are not allowed to proceed independently.
2. **Soak proof that validates benchmark claims**
   Exercise and soak must move together, or we are grading a lab artifact.
3. **Memory and learning refinements**
   Improve retrieval, learning, and recursive decomposition only after the
   proof surfaces are honest enough to detect real gains.
4. **Roadmap / compliance / platform truth cleanup**
   Important, but not allowed to outrank benchmark honesty.

## RCA-Derived Efficacy Scoring

- **Top concern: self-negating completion is contaminating benchmark truth**
  - repeated outputs of the form "I can't honestly claim success because the
    final verification still failed..." are no longer an isolated oddity
  - this is especially concerning because the current wording is emitted by the
    verifier finalization path, which means the failure class is at least
    partly framework-shaped rather than cleanly attributable to raw model
    capability
  - operating constraints for the fix:
    - a less-than-perfect truthful answer is better than silence or theatrical
      pseudo-refusal
    - `v1.0.7` should bias toward degraded truthful output when possible rather
      than collapsing near-answers into no-answer behavior
    - canned fallback responses are forbidden; the system must not paper over
      verifier/policy failures with stock text
  - `v1.0.7` should treat this as a release-critical benchmark and behavior
    hardening concern:
    - separate true model inability from verifier/finalization overreach
    - stop counting self-negating near-answers as ordinary clean model output
    - decide when the system should produce a degraded truthful answer versus a
      hard failure close
    - surface this pattern as its own RCA slice in benchmark and soak reports

- **Paired top concern: verifier sensitivity is likely manufacturing a large
  share of the plague**
  - current RCA evidence suggests the dominant retry trigger is
    `unsupported_certainty`, not outright contradiction or obvious hallucination
  - a large share of affected benchmark prompts are derivable or in-turn
    computable (`What time is it?`, `2 + 2`, basic coding facts), which means
    memory absence should be neutral rather than confidence-destroying
  - working hypothesis for `v1.0.7`:
    - memory should influence confidence triadically:
      - `-1`: retrieved memory contradicts the answer
      - `0`: memory is absent, irrelevant, or the result is derivable and
        therefore mutable rather than something expected to pre-exist in memory
      - `+1`: retrieved memory reinforces the answer
    - missing memory tiers must not be treated as proof that required evidence
      is absent
  - `v1.0.7` investigation tasks:
    - quantify how often verifier retry is driven by `unsupported_certainty`
      versus real contradiction/proof issues
    - separate memory-gap semantics from evidence-gap semantics
    - identify which benchmark classes are being over-penalized because
      retrieval or decomposition policy is too broad
    - simulate how many degraded benchmark rows would have survived under the
      trinary memory-confidence rule
  - planned correction sequence:
    - phase 1: apply ternary memory-confidence semantics, retest, and measure
      how much of the plague disappears
    - phase 2: investigate the remaining contra-indicated rows and determine
      what percentage is still verifier tuning versus other causes
    - phase 3: apply the most probable corrective path using evidence from
      phases 1 and 2 rather than intuition alone
  - acceptance criteria:
    - benchmark RCA can distinguish memory absence from real evidence failure
    - derivable prompts are not penalized simply because the system has not
      seen them before
    - verifier sensitivity is measured explicitly in scorecards and soak output
    - any remaining degraded outputs are attributable to real contradiction,
      proof burden, or subgoal failure rather than missing memory tiers

- **Post-observation reflection is the next architecture seam**
  - fresh RCA after the ternary memory correction now shows the dominant
    residual failure family is not raw model emptiness
  - the common pattern is:
    - first model attempt succeeds
    - tool call succeeds
    - another provider-facing inference call occurs
    - that later call fails with provider/protocol mismatch
    - the turn degrades into `empty response`
  - `v1.0.7` should therefore explicitly treat the live loop as the `TEOR`
    core inside the broader `R-TEOR-R` architecture:
    - leading `R`: retrieval memory before the turn acts
    - `TEOR`: think, execute, observe, reflect
    - trailing `R`: retention memory after reflection decides what the turn proved
  - post-observation reasoning is therefore a distinct architecture problem:
    - after authoritative observation, the system is meant to interpret,
      validate, refine, and decide whether further execution is justified
    - it is not meant to reopen broad execution inference by default
    - the canonical reflection input should be a `TOTOF` artifact:
      task, authoritative observed results, key tool outcomes, open issues, and
      bounded finalization instruction
  - investigation tasks:
    - define the exact contract of the reflect phase
    - define the canonical `TOTOF` reflective artifact and provider-specific
      renderers
    - define the canonical continuation artifact for
      `CONTINUE_EXECUTION`/reflect-returned-tool-call paths so continuation
      does not fall back to raw session replay
    - identify where current loop/pipeline behavior routes reflect work through
      ordinary inference semantics
    - distinguish execution success followed by reflection/protocol failure
      from genuine model inability
    - quantify which intent cohorts still require another model pass after tool
      observation and which should finalize deterministically from observed
      evidence
  - acceptance criteria:
    - successful tool-backed turns cannot be erased into emptiness by a later
      reflection/protocol failure
    - reflection continuation uses provider-safe canonical continuation state
      rather than raw assistant/tool-call history
    - RCA reports `execution succeeded, reflection failed` as its own failure
      class
    - benchmark empties are no longer dominated by provider/protocol mismatch
      after successful observation
    - source-backed code/refactor turns stop wandering through broad repo
      inventory and instead converge on authoritative source reads rooted in
      the actual current codebase

- **Correction burden as a first-class efficacy metric**
  - derive a per-model and per-turn correction-burden artifact from canonical
    RCA instead of treating all successful outcomes as equal
  - include at least:
    - normalization corrections
    - blocked malformed tool calls
    - guard retries
    - verifier retries
    - replay suppression
    - degraded final status
  - keep framework-owned corrections separate from model-owned error burden so
    we do not punish a model for our own defects
  - make the artifact reusable for:
    - benchmark scoring
    - benchmark evaluation itself
    - routing recommendations
    - operator guidance
    - future self-healing / recommendation logic
- **Use RCA to evaluate the benchmark, not just the models inside it**
  - benchmark output is not trustworthy if the harness, prompt set, rescue
    policy, or grading logic is systematically flattering weak behavior
  - `v1.0.7` should explicitly run RCA against benchmark design failures such
    as:
    - prompts that are too easy, too synthetic, or accidentally overfit to the
      harness
    - grading that rewards polished wrong answers or hides retrieval/tool
      misuse
    - rescue paths that make weak models look stronger than they are
    - cohort imbalance that overstates success on one intent class and masks
      failure on another
    - benchmark drift where we keep adding cases but never retire or relabel
      invalid ones
  - required RCA output:
    - benchmark-failure taxonomy
    - benchmark-case defect log
    - recommended cohort additions / removals / relabeling
    - explicit separation between model failure and benchmark-design failure
    - explicit separation between raw model failure and framework-shaped
      verifier/finalization failure
- **Make the benchmark surface prove real model efficacy, not just average synthetic prompt quality**
  - the current `models exercise` path is useful but still too synthetic:
    - static 35-prompt matrix
    - mostly one-dimensional average quality output
    - no explicit burden/correction view in the scorecard
    - generic scoring still rewards verbosity, structure, and intent-marker
      density even when a prompt's true contract prefers concise directness
    - targeted exercise reporting can silently fail to populate the historical
      comparison landscape, leaving a single freshly exercised model presented
      as `#1` in a meaningless one-row leaderboard
  - `v1.0.7` should add:
    - medium-lift prompt-contract scoring:
      - extend each exercise prompt with a scoring contract
      - evaluate contract satisfaction before generic intent/style heuristics
      - encode whether concision is preferred, neutral, or harmful
      - encode whether tool use is required, optional, or contraindicated
      - treat format strictness as prompt-specific rather than global
      - preserve rescoring of historical exercise rows when raw prompt/response
        artifacts are complete
    - explicit benchmark cohorts:
      - canonical synthetic matrix
      - live-source / adversarial cohort drawn from real failure classes
      - memory-and-tool-bearing cohort that exercises retrieval + tool choice
    - reporting that separates:
      - raw quality
      - correction burden
      - latency
      - pass / degraded / rescued / failed
      - model-attributable latency from whole-turn/pipeline latency
    - recommendation output that can say:
      - "good final answer but high rescue burden"
      - "good on conversation, weak on memory recall"
      - "acceptable only with tool-free tasks"
  - acceptance criteria:
    - concise contract-satisfying answers are not penalized for omitting
      irrelevant intent markers or extra prose
    - verbosity is not rewarded by default
    - historical exercise rows can be rescored under the new prompt-contract
      rubric when persisted artifacts are complete
    - new `exercise_results` rows persist canonical `turn_id` so scorecards,
      traces, and RCA can join without timestamp heuristics
    - benchmark callers establish `turn_id` before pipeline execution so failed
      or degraded turns keep the same RCA join key as successful turns
    - empty reflection after tool observation is finalized from observed
      evidence or recorded as an error, never persisted as a blank answer
    - CLI benchmark transport treats local API problem-details `detail` fields
      as errors instead of scoring them as empty model responses
    - live CLI exercise summaries and persisted scorecards either use the same
      denominator semantics or label any intentional difference explicitly
    - `Model Comparison` and per-intent latency summaries use composite model
      inference latency, while phase timing retains total pipeline latency for
      RCA
    - partial intent exercise runs merge the freshly exercised intent slice
      into historical scorecard evidence instead of replacing the model's
      untouched intent scores with zeroes
    - exercise runs persist enough artifact detail to compare models by intent,
      latency, and correction burden together
    - scorecards stop collapsing everything into one average quality number
    - low-observation models are clearly labeled as under-evidenced rather than
      treated as confidently ranked
    - model comparison rows display evidence coverage so one-intent averages
      cannot masquerade as full seven-intent baselines
    - exercise rows persist row-level outcome class so timeout, empty-response,
      slow-valid, transport, and quality-gate cases are not collapsed into one
      generic failure bucket
    - benchmark RCA can explain whether a disappointing score came from the
      model, the task, the grader, or the rescue policy
    - repeated self-negating completion rows are called out explicitly and do
      not silently count as ordinary model prose
    - explicit-model exercise runs still render the full previously exercised
      comparison set, with freshly exercised models highlighted rather than
      isolated in a fake one-row ranking
    - verifier/finalization changes do not replace bad outcomes with canned
      stock responses; they must preserve task-specific truthful output or an
      explicit task-specific failure explanation
- **Promote coding evaluation from prose theater to artifact-grounded grading**
  - the current coding column is still too prose-shaped to support strong
    claims about coding effectiveness
  - `v1.0.7` should raise the standard to something closer to how a teacher
    would grade concrete code work:
    - does the submitted code parse / typecheck / compile for the target
      language when the prompt asks for runnable code?
    - given a bounded array of inputs, does it produce the correct outputs?
    - only after artifact correctness, score explanation quality, edge-case
      awareness, and implementation clarity
  - active implementation scope:
    - bounded artifact extraction for coding prompts that request runnable code
    - compile / parse / typecheck evaluation where the language/runtime permits
    - fixed input/output correctness evaluators for bounded cases
    - prompt-contract fallback only for coding prompts that are genuinely
      explanation-only rather than artifact-bearing
  - acceptance criteria:
    - coding rows are not graded primarily by prose richness
    - an uncompilable or incorrect code artifact cannot outscore a correct,
      concise artifact because the explanation is more polished
    - coding reports surface whether the artifact was executable, correct on
      bounded cases, both, or neither
- **Source-backed code execution hardening**
  - current-repository refactor/fix/debug turns need a distinct focused
    execution envelope rather than the generic heavy `intent=code` path
  - active implementation scope:
    - detect source-backed code surgery prompts separately from greenfield code
      generation or explanation-only code prompts
    - anchor those turns on the actual current source root
    - prioritize authoritative source-read / bounded inspection tools before
      broad inventory or speculative workspace scans
    - treat productive read/list/glob steps as progress for this profile so
      the loop converges instead of misclassifying bounded source discovery as
      churn
    - keep retrieval neutral unless the prompt carries explicit continuity or
      evidence cues
  - acceptance criteria:
    - expert refactor prompts no longer default to broad `/users/.../code`
      inventory when the current repository is the obvious target
    - source-backed code turns locate and read authoritative source files
      before finalizing or giving up
    - repo-grounded bounded inspection counts as convergence progress rather
      than exploratory churn for this profile
- **Deferred high-lift benchmark evaluator program**
  - keep on the roadmap for the next benchmark-architecture pass:
    - prompt-class semantic evaluators that reason over stronger equivalence
      classes than keyword markers
    - structured-output-aware grading where the prompt contract demands it
    - tool-trace-aware evaluators that inspect whether the model used the
      correct execution path, not just whether the final prose sounded right
    - richer benchmark validity labels when semantic evaluation cannot be
      decided from the available artifacts
    - broader multi-language artifact evaluators and stronger correctness
      oracles beyond the bounded compile/test slice above
  - this program should begin only after the active coding-evaluator slice and
    scoring-contract seam are stable enough to support honest rescoring

## Benchmarking Program Hardening

- **Refine the exercise matrix into something release-grade**
  - current `models exercise` already supports intent slices, iterations,
    warm-up, run persistence, and scorecards
  - it still needs stronger release-grade proof surfaces:
    - multiple named cohorts with explicit purpose
    - historical-vs-fresh delta reporting
    - confidence intervals or at least variance/dispersion reporting for
      repeated iterations
    - regression callouts by intent class, not just overall ranking
- **Concrete `v1.0.7` benchmarking additions**
  - add a benchmark cohort registry instead of one implicit prompt matrix
  - add "release gate" benchmark views:
    - routing cohort
    - memory recall cohort
    - tool-use cohort
    - coding cohort
  - add result summaries for:
    - best model per intent
    - most improved / regressed since prior run
    - models that pass quality but fail latency
    - models that pass latency but fail burden / rescue criteria
    - benchmark cohorts whose own validity is in doubt under RCA review
    - models/categories with high self-negating completion incidence
  - add first-class phase telemetry:
    - retrieval-memory timing
    - think / primary model-inference timing
    - execute / tool time
    - observe normalization time
    - reflect timing
    - validate / verifier timing
    - retry / continuation overhead
    - total wall-clock timing
  - add a benchmark truth rule:
    - routing recommendations are not allowed to rely only on average quality;
      they must include burden and evidence coverage too
    - benchmark-driven decisions are not allowed to rely on cohorts with
      unresolved RCA defects
    - benchmark-driven decisions are not allowed to ignore epidemic
      self-negating completion patterns as if they were harmless tone
    - benchmark summaries are not allowed to present aggregate quality without
      making clear whether failures contributed zeroes or were excluded from an
      intent average
    - benchmark-driven decisions are not allowed to treat verifier sensitivity
      as if it were clean model weakness
  - add a benchmark liveness rule:
    - exercise timeout policy is prompt-aware rather than one flat cloud
      timeout
    - operator/CLI wait budget is not the same thing as the authoritative
      server execution budget for long `T-E-O-R` rows
    - provider/model timeout is applied per inference call and resets after
      each completed model response; total row wall-clock remains separately
      measured so framework latency is not charged as model failure
    - exploratory baselines use a high finite ceiling for observation, while
      release gates use stricter SLO views
    - trace/diagnostic persistence survives client timeout or disconnect under
      a bounded detached persistence budget once a turn has started

## Soak Proof Expansion

- **Soaks should prove the claims we are making about efficacy and routing**
  - current soak surfaces exist, but they do not yet serve as a first-class
    proof system for the release thesis
  - `v1.0.7` should add named paired soaks that prove:
    - baseline agent efficacy
    - memory-bearing efficacy
    - routing efficacy under real intent mix
    - cache-on vs cache-bypassed behavior where relevant
- **Concrete `v1.0.7` soak additions**
  - extend the behavior soak with explicit result classes:
    - pass
    - degraded-but-corrected
    - failed
  - add a memory-bearing soak slice that requires:
    - correct retrieval
    - correct `search_memories` / `recall_memory` tool choice when needed
    - no confabulated memory answer when retrieval evidence is absent
  - add a routing-efficacy soak slice that requires:
    - selected model and actual execution model both recorded
    - route rationale attached
    - fallback / escalation / rescue paths visible in the report
  - add paired soak output that can answer:
    - did the model succeed?
    - how much correction was needed?
    - did memory actually help?
    - did routing choose a sensible model for the task class?
    - was the final answer degraded truthfully, or did verifier/finalization
      collapse into self-negating pseudo-failure language?
  - release proof requirement:
    - we do not claim benchmark or routing improvement unless the soak output
      and the exercise output both move in the same direction

## Memory And Learning Refinements

- **Add the memory improvements we already know are warranted**
  - current memory architecture is strong, but the spec still admits known
    remaining work:
    - episodic-to-semantic promotion discipline / expansion
    - telemetry-gated LIKE retirement
    - stale episodic FTS on UPDATE
  - the current learning layer also remains crude in places:
    - quality tracking is still a sliding average
    - procedure learning is wired, but simplistic and weakly surfaced
    - promotion/learning outputs are not yet tied back into benchmark proof in
      a disciplined way
  - the semantic-collapse-defense thesis from
    `docs/plans/semantic-collapse-defense-roadmap.md` should be treated as an
    explicit exploration/proof item for `v1.0.7`, not just background reading
  - the Recursive Language Models paper
    (`https://arxiv.org/html/2512.24601v1`) should also be treated as an
    explicit `v1.0.7` investigation input for memory, retrieval, and learned
    recursive decomposition rather than left as loose reading material
- **Concrete `v1.0.7` memory / learning work**
  - retrieval truth:
    - close the episodic FTS update-trigger gap so edited episodic content
      cannot silently drift from search results
    - make LIKE-retirement evidence explicit enough to remove more fallback
      paths tier by tier instead of leaving them indefinitely dormant
    - add an explicit semantic-collapse exploration slice:
      - corpus-scale retrieval metrics that track score spread / narrowing
      - proof that reranking and fusion continue to separate strong from weak
        evidence as corpus size grows
      - targeted synthetic or replayable large-corpus cases so we are not
        inferring safety from dev-scale data alone
    - add an explicit RLM exploration slice:
      - evaluate whether recursive retrieval / summarize / re-query loops
        improve memory-bearing tasks without blowing up latency or confabulation
      - identify whether any of the paper's useful structure can be expressed
        with today's planner, subagent, or tool loop instead of requiring a new
        model architecture first
      - produce a short list of immediate practical applications, if any, such
        as recursive query decomposition, staged evidence distillation, or
        learned memory summarization passes
  - learning truth:
    - strengthen episodic-to-semantic / distilled-learning reporting so we can
      prove what was promoted, why, and with what support threshold
    - expose learned procedure / learned-skill outcomes with clearer operator
      observability instead of burying them in background persistence
    - replace or augment naive moving-average quality tracking with
      recency-aware / evidence-aware aggregation that does not flatten sharp
      regressions into a harmless mean
    - if the RLM investigation yields an obvious low-risk improvement, ship the
      narrow practical slice in `v1.0.7` instead of stopping at literature
      review
  - retrieval efficacy:
    - add explicit memory ROI proof per intent class, not just coarse aggregate
      analytics
    - prove that memory-bearing tasks improve with retrieval and degrade
      safely when evidence is absent
  - release proof requirement:
    - no memory-improvement claim without paired exercise + soak evidence on
      memory-bearing tasks

## Windows Confidence Expansion

- **Add Windows to the testing matrix in phased confidence order**
  - phase 1:
    - add `windows-latest` to the CI compile/unit matrix
    - keep the goal narrow: catch compile breaks, path-separator drift,
      executable-suffix mistakes, and obvious platform assumptions
  - phase 2:
    - add a Windows smoke/install gate so release confidence includes basic
      runtime and upgrade truth on Windows, not just compilation
  - phase 3:
    - add meaningful Windows behavioral/soak coverage only after the shell,

      those signals trustworthy
  - required audit surfaces:
    - shell/script Unix assumptions (`bash`, `sed`, `mktemp`, `sha256sum`,
      signals, process groups)

    - daemon control / install / upgrade paths
    - packaging and installer verification

## CI Test-Surface Hygiene

- **Split and profile the slow CI test surface, especially on ubuntu under `-race`**
  - current observed evidence from the `v1.0.7` release path:
    - macOS test job: `8m 12s`
    - ubuntu test job: `20m 25s`
  - identify the worst package-level outliers in CI timing, with immediate
    attention to:
    - `internal/api/routes`
    - `internal/agent/memory`
    - `internal/db`
    - `internal/pipeline`
  - produce a top-slowest-tests inventory from the ubuntu runner instead of
    treating the wall-clock gap versus macOS as generic runner noise
  - move integration-like and soak-like cases out of broad package test runs
    when they do not belong in ordinary unit/package gates
  - preserve deterministic unit coverage in-package while narrowing the heavy
    environment-sensitive cases into explicit targeted jobs
  - success criteria:
    - ubuntu/macOS timing differences are explainable by scoped workloads, not
      accidental package bloat
    - ordinary PR test jobs stop carrying hidden integration/soak weight
    - CI duration is reduced without weakening release-confidence coverage
