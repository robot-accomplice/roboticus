## Architecture: Externalized Model Profiles

### Problem

Model-specific parsing and formatting (tool call format, response extraction, stop tokens, system prompt position) is currently hardcoded in Go. Adding support for a new model requires code changes and recompilation. This creates binary bloat as the variety of models grows and prevents codeless updates for new model support.

### Specific examples from v1.0.4 soak

- `qwen2.5:32b` outputs tool calls as raw JSON text `{"tool_call":{"name":"bash","params":{...}}}` instead of structured function calling — the pipeline's `ParseToolCallsFromText` handles this but it's a compiled fallback, not a declared behavior
- Different models use different tool call formats: OpenAI function_call, Anthropic tool_use, raw JSON text, XML tags
- Stop token handling varies by model
- System prompt positioning (before user vs interleaved) varies

### Solution: Model Profile Registry

Model profiles as external data files (TOML/JSON) loaded at runtime from `~/.roboticus/models/` or fetched from the site registry:

```toml
# ~/.roboticus/models/qwen2.5-32b.toml
[identity]
name = "qwen2.5:32b"
provider = "ollama"
family = "qwen"

[tool_calling]
format = "json_text"           # "openai" | "anthropic" | "json_text" | "xml_tags" | "none"
extraction_pattern = '{"tool_call"'  # marker for text-based extraction
stop_tokens = ["<|im_end|>"]

[context]
system_prompt_position = "first"  # "first" | "interleaved" | "none"
max_context_tokens = 32768

[formatting]
chat_template = "chatml"       # "chatml" | "llama" | "mistral" | "raw"

[performance]
default_timeout_seconds = 300
cold_start_seconds = 30
supports_streaming = true
```

The LLM client loads the matching profile when connecting to a model. `ParseToolCallsFromText` becomes data-driven — the profile's `tool_calling.format` determines which parser runs.

### Benefits

- New model support = new profile file, no recompile
- Profiles hosted on `roboticus.ai/registry/models/` alongside themes/plugins
- Users can override profiles locally for fine-tuning
- Binary stays lean — parsing logic is generic, behavior is data

### Migration path

1. Extract current hardcoded behaviors into default profiles for known models
2. Add profile loader to LLM client initialization
3. Make `ParseToolCallsFromText` dispatch based on profile format
4. Ship profiles on the site registry
5. Add dashboard UI for model profile management

## Soak findings to address

### filesystem_count_only timeout (RESOLVED in v1.0.4)
- 32B local model timed out on recursive file counting across large directory trees
- Root cause: soak script defaults were too aggressive (240s HTTP timeout, 120s latency limit) — soak tests are long-running by design and need extraordinary upper bounds
- Fix: raised defaults to 1800s HTTP timeout, 900s latency, added per-scenario `max_latency_s` override (filesystem_count_only gets 1800s)
- v1.0.5 model profiles will further improve this by declaring cold-start times per model

### Model output format variance
- Some models (especially local quantized) output tool calls as raw text
- v1.0.4 added cache guards to prevent caching these incomplete responses
- v1.0.5 model profiles will make this a declared behavior per model
