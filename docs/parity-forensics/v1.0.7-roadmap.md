# v1.0.7 Parity Roadmap

This document is the authoritative execution inventory for parity work in
v1.0.7.

Rules:

- Every unique remaining parity gap gets exactly one roadmap entry.
- No remaining gap is allowed to exist only as prose in a system doc, release
  note, audit log, or old gap register.
- A roadmap item is not `closed` until the linked closure proof exists in code,
  tests, or runtime evidence.
- No new deferrals are allowed without explicit operator endorsement.

## Status Definitions

- `not started`: acknowledged and scoped, no remediation landed
- `in progress`: remediation or proof work is actively underway
- `blocked`: cannot close without an external dependency or explicit operator
  decision
- `closed`: remediation and closure proof both exist
- `explicitly deferred by operator`: only valid with explicit operator
  endorsement

## Notes

- `PAR-002` through `PAR-006` intentionally split the old “missing Rust
  operational tool families” deferral into distinct lifecycle seams. `PAR-002`,
  `PAR-003`, `PAR-005`, and `PAR-006` are now closed separately so roster,
  composition, task lifecycle, and orchestration cannot drift under one vague
  bucket.
- `PAR-010` and `PAR-011` intentionally split verifier depth into two closure
  targets so contradiction handling and proof sufficiency do not get collapsed
  into one vague “make verifier better” item.
- `PAR-012` through `PAR-014` are architecture-led retrieval closures and are
  expected to touch both parity docs and the retrieval pipeline.
