# v1.0.7 Release Notes

Status: ready for develop merge pending green CI

## What This Release Is For

v1.0.7 is focused on three operator-facing outcomes:

1. clearer incident diagnosis when the agent behaves badly
2. more reliable behavior from the orchestrator and subagents
3. better routing decisions, especially on constrained local hardware

This is not a cosmetic release. It is intended to make the system easier to trust, easier to debug, and less likely to waste time on avoidable failures.

It is also being run as an explicit behavior-hardening program:

- repeated firsthand failures and operator reports now enter through canonical RCA
- each unique framework-owned failure class gets its own roadmap item
- obvious repeated failures are the target, not every speculative weakness
- fixes are only considered complete when architecture, documentation, RCA,
  regressions, and live behavior all agree
- model registry truth is being tightened as one system, so routing,
  execution, and RCA cannot disagree about which provider/model pair was
  actually selected and used
- low-level/core boundaries are being re-tightened so storage-layer seams do
  not depend upward on agent packages for generic helpers, and oversized
  route-admin connector files are being split back into bounded connectors
  without losing unified pipeline-path enforcement across that connector
  surface
- release/deployment truth is also being tightened so release-shaped binaries
  no longer depend on a miswired CLI ldflags symbol that would leave
  `roboticus version` reporting `dev` after CI/release packaging
- release-branch hygiene is also being tightened so dead helper seams and
  mechanical lint drift are removed before merge instead of being waved through
  as “just CI noise”
- release-test hygiene is also being tightened so async delegation persistence
  in tests cannot outlive `TempDir` cleanup and manufacture false CI failures
  on the release branch
- GitHub Actions workflow hygiene is also being tightened so release/CI jobs use
  Node 24-capable action majors instead of carrying known deprecation warnings
  on the release branch
- artifact/download actions and the security scan path are also being tightened
  so CI no longer vendors Node 20-era runtimes through `upload-artifact`,
  `download-artifact`, or the composite `govulncheck` wrapper
- release security truth is also being tightened so the repository-declared Go
  patch line matches the fixed standard-library floor required by
  `govulncheck`, instead of letting local `go` auto-upgrade mask a vulnerable
  CI toolchain
- MCP stdio failure diagnostics are also being tightened so stderr-tail
  capture waits on transport-owned collector completion instead of a timing
  heuristic; slower CI runners no longer drop the actionable tail marker from
  a flooded child stderr stream
- MCP stdio diagnostic tests are also being tightened so the CI gate uses
  deterministic in-process helper children instead of shell pipelines and
  `/bin/sh` redirection semantics
- WhatsApp formatter fuzz stability is also being tightened so malformed
  delimiter floods no longer send the release-gate fuzz target into quadratic
  inline-markup scans on slower runners
- release execution is also being tightened so the canonical `Release`
  workflow can be dispatched explicitly against an already-created tag when the
  original tag-push event fails to enqueue, instead of forcing a manual
  artifact side path or a second fake release tag
- release publication is also being tightened so the rerunnable path no longer
  depends on `softprops/action-gh-release` inferring tag context from
  `github.ref_name`; the workflow now publishes and updates releases explicitly
  through `gh` using the requested release tag as the single authority
- release workflow notifications are also being tightened so repository
  dispatches use explicit `gh api` calls instead of third-party action context,
  CI security scanning pins `govulncheck` to a declared version instead of
  floating `latest`, and the workflow now performs a final self-evaluation plus
  email report instead of assuming a green publish step proves release success
- verifier coverage is also being corrected to derive one canonical subgoal set,
  so the framework stops manufacturing `2/3` coverage failures by counting both
  the whole prompt and its conjunction-split parts as separate requested goals

## What Operators Should Notice

### Better visibility when a turn goes wrong

The runtime now records much better turn-level diagnostics. When a turn is slow, degraded, or wrong, the system is increasingly able to say:

- what kind of task it thought it was handling
- how much context and tool surface it assembled
- which model/provider path it chose
- whether it was still waiting on the first attempt or stuck in retries
- what fallback or retry behavior occurred

This work is aimed at making failures explainable instead of opaque.

That explanation trail now also has to include the machine state the turn ran
under. Host CPU, memory, swap, and relevant process RSS are being promoted
into the same benchmark and RCA artifacts so operators can tell the difference
between a weak model and a saturated machine.

Operators can now see that explanation trail in the WebUI flow itself instead of reconstructing it from logs:

- the flow view can read the canonical turn diagnostics artifact
- a default macro view now presents a desktop-first left-to-right decision flow grouped into task, envelope, routing, execution, recovery, and outcome
- a visible detail toggle can expose the chronological diagnostic event timeline when deeper RCA is needed
- the old stage flow and RCA split are being collapsed into one bounded decision flow instead of parallel surfaces
- macro blocks stay compact, dense turn state is moving into one narrow status banner instead of a row of large tiles, and turn conclusion plus health can move to a separate bottom banner when space is tight
- macro nodes are being stripped down to one useful visible signal each, with timing as the default and routing as the exception because the chosen model matters more than repeated internal stage names
- verbose text is moving into a true floating tooltip layer plus explicit detail mode
- the flow surface is expected to stay inside the real usable main-pane width, accounting for the sidebar and other persistent chrome, or expose explicit scrolling when it cannot
- if a turn predates canonical diagnostics, the flow now needs to say that explicitly and fall back to a trace-only narrative instead of silently dropping operators back into an unlabeled stage dump
- pipeline traces and canonical diagnostics now need to share the same live turn identity, so fresh turns with RCA data do not get mislabeled as trace-only fallback in the operator UI
- repeated execution is now being promoted into the flow itself with explicit repeat markers on the affected steps
- retry causality is being surfaced directly in the UI: which guard or verifier fired, the retry reason, whether the same model/provider was retried, whether fallback was used, and the final outcome
- the conclusion banner is being upgraded from a telemetry acknowledgment to a real interpretation of what the evidence implies
- the persisted turn-diagnostics summary itself is also being upgraded so downstream UI and RCA consumers are not forced to reinterpret placeholder boilerplate
- flow blocks are gaining outcome coloring from the same RCA evidence: green for clean, yellow for concern, red for broken
- the dense RCA top banner is being aligned to those same semantics as well, including thresholded latency / pressure coloring and an explicitly aggregate health reading rather than a disconnected summary judgment
- the top banner metrics are also gaining explanatory hover/focus tooltips so values like `degraded`, `high`, or host-pressure readings are interpretable without leaving the flow
- stale trace-only fallback overlays are being torn down when the active session or expanded turn changes so old fallback content does not visually contaminate fresh canonical-diagnostics turns
- benchmark and RCA analysis are also starting to capture raw provider request/response envelopes at the same tool-normalization seam, so framework/tool-format defects can be compared directly against provider documentation instead of inferred from scored content alone
- applied-learning retrieval and reuse capture are being promoted into the same operator-facing RCA flow, so the system can explain when it consulted prior successes/failures and when a turn produced reusable procedural knowledge
- the soak harness itself is also being tightened so clone/fresh scripted lanes fully own and tear down their managed `serve` process groups; prompt-compression comparisons no longer need to sit behind stale orphaned lane servers or wait out a phantom port-release timeout between baseline and compressed runs
- focused filesystem inspection is now being tightened around structured
  evidence instead of plain text, so the loop and RCA can tell the difference
  between a useful file-system observation and an empty miss without re-parsing
  ad hoc strings
- direct-execution turns are no longer supposed to die on the vine with
  `let me check...`-style future-action filler; that class is now being treated
  like placeholder output unless a real tool call or authoritative evidence
  followed
- the runtime context seam is also being tightened so the agent sees the
  effective sandbox truth it actually runs under rather than a vague “security
  policy” promise backed only by a partial path list
- reusable learning capture is now being tightened so final turn status and
  verifier outcome remain authoritative for outcome polarity; degraded or
  verifier-failed turns are not supposed to be promoted as successful lessons
  just because the underlying tool calls happened to succeed
- once the pipeline has already assembled current-turn retrieved evidence and a memory index, the prompt now treats that injected evidence as the first memory authority for the turn; follow-up `recall_memory` / `search_memories` calls are reserved for real evidence gaps instead of being triggered unconditionally
- source-backed authoring is being tightened so `read_file`-style source reads
  are first-class focused-authoring tools rather than “unknown” semantics, and
  verifier retries that identify `source_artifact_unread` now carry structured
  corrective guidance that prefers authoritative source reads over memory churn
- source-backed exact artifact turns are also being kept on the focused
  authoring path instead of being misclassified into the generic heavy/code
  envelope just because they mention JSON, runbooks, or a source file
- the bounded selected tool surface is now being enforced as one authority
  across both prompt guidance and execution, so a tool that was pruned off a
  turn is no longer supposed to linger in generic prompt instructions or
  execute anyway through the global registry
- artifact filenames like `runbook` or `policy` are no longer supposed to
  trigger authority-mutation mode by themselves, so focused file/note authoring
  does not silently widen back into a broader operational tool surface
- prompt-declared source artifacts are now being tightened into protected
  read-only turn resources so a model cannot satisfy a required source read by
  overwriting the source file with an artifact-writing tool
- source-backed focused authoring turns now also pin an authoritative
  `read_file`-style tool on the selected surface, and `source_artifact_unread`
  stays execution-critical after progress instead of being suppressed as merely
  narrative-quality verifier noise
- sandbox path authorization is now being corrected for symlinked roots and
  future child write targets, so authoring paths under temp/workspace aliases
  like `/var/...` versus `/private/var/...` do not drift between “allowed” and
  “denied” depending on whether the leaf already exists
- tool-bearing routing now uses observed `TOOL_USE` evidence as a real
  selection boundary when tools are present, so under-evidenced models are not
  merely narrated as “ignored” while still being selected first
- rebuildable derived SQLite corruption now has one database-owned repair path:
  if damage is confined to observability tables or `memory_fts` internals, the
  store backs up the damaged files, rebuilds the affected derived structures,
  backfills FTS from authoritative memory tables, and retries the interrupted
  operation instead of letting telemetry corruption poison the whole daemon
- focused filesystem inspection now resolves home-directory aliases such as
  `~` and “home folder” through the same target-authority seam as absolute
  paths, using the live current-user home directory instead of machine-specific
  assumptions, so inspection requests like “give me the file distribution in the
  folder ~” no longer collapse into empty conversational acknowledgements
- focused scheduling now treats explicit cron creation as a known direct
  procedure while using session continuity to resolve shorthand aliases like
  `quiet ticker`, so compressed-history scheduling turns no longer widen into
  procedural churn just because the operator is reusing a prior definition

That explanation path is also getting deeper inside verification itself:

- contradiction handling is moving from a boolean hint to a structured evidence artifact
- claim audits are starting to show what proof each high-risk claim required
- missing-proof diagnostics are becoming explicit enough to explain why the system revised or blocked a response
- verifier traces are being promoted from raw claim maps toward operator-legible summary counters for contested claims and proof gaps

### Better behavior on lightweight conversational turns

Simple conversational turns are being handled more cleanly:

- lighter prompt envelopes
- fewer unnecessary tools
- less irrelevant operational/status leakage
- less pointless retry churn

This means short social or conversational prompts should feel less heavy-handed and less error-prone than before.

The first explicit behavior-hardening slice is now also attacking repeated
same-route no-progress churn:

- when the same route keeps returning structurally identical no-progress
  responses with no new tool progress, the loop now terminates instead of
  silently spinning until generic idle handling
- canonical RCA now records that termination cause explicitly, so operators can
  see that the framework stopped repeated same-route churn on purpose rather
  than infer it from a shorter attempt count
- tool execution truth is also being tightened: `tool_calls` and RCA counters
  now need to be written from the loop’s execution events instead of inferred
  later from session history, so a tool-bearing turn cannot claim zero tool
  calls after real tool use
- persistent-artifact authorship is now being tightened as a separate seam:
  note/document/file creation turns must privilege artifact-writing tools over
  authority-layer memory mutation, and the framework is no longer allowed to
  accept “created note” or “saved file” claims unless the turn carries
  matching artifact-writing evidence

The same control-plane tightening now applies to simple direct tasks:

- `simple` + `execute_directly` action turns are no longer widened into heavy autonomous tool-bearing turns by the `task` label alone
- retrieval is no longer enabled for imperative turns by default; action turns now need explicit continuity/context cues before memory gets pulled in
- the first pass for simple direct work now stays on a focused execution envelope with a bounded tool surface instead of a heavy autonomous loop
- bounded multi-artifact authoring turns are now being pulled into that same focused execution path instead of falling back to `complex` specialist work just because the request names more than one explicit artifact
- verbose single-step authoring requests no longer get upcast into a broader execution envelope on word count alone, so explicit note/document/file creation can still stay on the focused tool surface

This is specifically aimed at the bad “simple task became a 10-attempt heavy turn” failure class.

This hardening is also closing a more systemic drift class:

- note/document authoring turns no longer get upcast into coding intent just because a filename or heading contains lexical noise like `test`
- DB-enabled skill files are now loaded into the live runtime from their authoritative source paths even when a configured skills directory is wrong or incomplete
- capability fit is now allowed to recognize hyphenated and description-backed skill concepts like `obsidian-vault` instead of treating `obsidian` and `vault` as missing just because there is no exact single-token name match
- when Obsidian is enabled but `vault_path` is blank, the runtime now auto-detects a real workspace-local vault instead of leaving the product in a half-enabled lie

### Better personality setup interviewing

The setup interview is being tightened so it converges on behavior more deliberately instead of treating personality selection as a shallow one-pass vibe exercise.

Current improvements:

- before the interview starts, the operator is now explicitly prompted to picture the kind of assistant they want, including a character from a book, film, or television show or any other concrete archetype that captures the desired behavior
- the first explicit interview question is now always the agent's name
- when the operator supplies a reference identity, the interview is now allowed to infer likely traits from that identity and use them as provisional seed assumptions
- those seeded traits are not treated as silent truth; the interview must surface them explicitly and ask the operator to confirm, reject, or refine them
- the interview is now instructed to ask broader and sometimes repeated behavior questions in different wording when important traits remain vague, contradictory, or underspecified
- the fallback opening path now preserves the same contract instead of collapsing to a generic name-only opener when the LLM is unavailable
- the interview contract is now centralized in one shared prompt/opening seam so API wording, degraded fallback behavior, and prompt-level interview guidance do not drift apart

This is not just copy polish. It is intended to produce better `OS.toml`, `FIRMWARE.toml`, `OPERATOR.toml`, and `DIRECTIVES.toml` outputs by improving the quality of the intake itself.

### Explicit Obsidian vault authoring

Obsidian support is no longer expected to work by prompt hint and generic file-tool inference alone.

Current improvements:

- when Obsidian is configured, the live runtime now registers an explicit `obsidian_write` tool
- system prompt guidance now points the model at that named vault-authoring capability
- vault note creation/update now has explicit runtime semantics instead of relying on the model to infer that a generic `write_file` call is acceptable
- workspace-local vaults can now be auto-detected when Obsidian is enabled but the path was left blank, so the runtime can converge on the same truth the operator sees in the workspace layout

This makes vault authoring much more honest as a product capability and reduces the gap between what the operator sees in skills/config and what the runtime can actually do.

Current hardening in this seam is broader than one new tool:

- DB-enabled skills, runtime skill loading, capability fit, prompt guidance,
  and tool registration are being forced to agree on one runtime truth instead
  of drifting across separate inventories
- Obsidian-related capability claims are being tightened so the UI, prompt,
  and live agent do not disagree about whether vault authoring is actually live
- simple note/document authoring turns are being protected from accidental
  upcasts into heavy coding envelopes caused by lexical noise in filenames or
  titles
- focused note/document/file authoring turns now use a capability-scoped tool
  profile instead of inheriting the generic operational always-include set, so
  the live request surface collapses to artifact-writing plus only the minimum
  runtime context actually needed for the turn
- once a turn has already completed a real tool-backed action, post-success
  retries are now adjudicated by one central policy: narrative-only concerns
  remain visible in RCA, but they no longer get to spend another model attempt
  unless the defect is execution-critical
- once a side-effecting tool call has already succeeded, the loop now suppresses
  duplicate replays of the same protected effect inside the same turn instead
  of risking a second non-idempotent mutation, even when the retry changes raw
  arguments but still targets the same artifact/resource
- replay protection is now being promoted into canonical diagnostics and the
  operator RCA surface as an explicit protected-resource fact rather than
  remaining hidden inside loop internals
- when the framework is procedurally uncertain how to carry out a task, task
  synthesis and perception can now trigger applied-learning retrieval from
  procedural and episodic memory so prior successes and prior failures are both
  available before the model starts exploring
- once that retrieval has already happened, the model is no longer instructed
  to re-search memory by default “even if injected memories are present”; the
  injected current-turn evidence and memory index are now the first authority,
  and follow-up memory search is reserved for genuine evidence gaps
- novel procedural experiences are now being carried forward with explicit
  outcome semantics in the learning pipeline, so repeated successes, failures,
  and mixed results can be promoted for future reuse instead of dying as
  one-off episode summaries
- those pre-inference learning decisions and post-turn reuse outcomes are now
  being promoted into the canonical RCA summary/flow as first-class facts,
  instead of staying buried in detail-only diagnostic events or memory rows
- direct execution turns are also starting to terminate on exploratory
  read-only tool churn: when a turn that should `execute_directly` keeps
  spending successful runtime-context / workspace / capability / task
  inspection / memory-read calls without crossing into real execution
  progress, the framework now records that as an explicit RCA failure class
  instead of letting the loop quietly burn tool calls forever
- artifact-writing tools are being upgraded from byte-count acknowledgments to
  typed proof emitters: successful file/note/document writes now carry one
  structured artifact-proof payload with path, bytes, content hash, and
  exact-content evidence when the written artifact is small enough to retain
  safely. That proof is being preserved through session tool-result metadata
  and projected into guard/verifier logic so direct authoring turns do not
  degrade into “cannot guarantee integrity” after the framework already has
  exact write evidence
- explicit exact-content authoring requests are now being promoted into one
  shared expected-artifact spec artifact at the prompt boundary, so turn sizing
  decomposition / turn-policy / verifier logic consume the same structured
  expectation instead of rediscovering it through separate heuristics. Embedded
  artifact bodies are no longer supposed to inflate bounded authoring into
  specialist work, and when the written artifact content does not match the
  requested exact content, that mismatch remains execution-critical even after
  tool-backed progress
- the exact-artifact parser now treats equivalent directive forms such as
  `containing exactly` and `with content` as the same prompt-boundary
  expectation, so verifier certainty checks are not allowed to miss exact
  artifact proof just because the operator phrased the request differently
- relative artifact names in those exact-content prompts now inherit an
  explicitly named container directory from the same request, so exact proof is
  compared against the real authored path instead of a detached bare filename
- trailing post-write/reporting instructions are now excluded from the last
  artifact body instead of being silently treated as part of the requested
  exact content
- verifier-triggered rewrites are now re-verified before finalization, so a
  failed retry cannot bypass exact-content verification and end as an
  operator-facing false success claim
- exact-artifact verification is being tightened into a symmetric check: it
  now must reject invented extra-file claims in the response and unexpected
  extra writes at the tool boundary instead of only proving that the required
  files conformed
- prompt-boundary artifact parsing must now also distinguish source/input
  artifacts from expected outputs, so truthful references to files the turn was
  told to read do not get misclassified as invented output claims
- structured tool I/O now passes through one central normalization seam
  before execution and observation instead of leaking malformed provider/model
  payloads directly into tools; repairable malformed arguments are normalized
  with explicit RCA evidence, while malformed arguments without a qualified
  transformer are rejected honestly at the tool boundary instead of degrading
  downstream into verifier retries or vague “tool use failure” impressions
- assistant `tool_calls` history is now treated as immutable provider/RCA
  truth instead of sharing one backing slice with mutable pending-call state,
  so consuming one tool result can no longer silently rewrite the tool-call
  plan sent back to the provider on the next inference pass
- explicit unknown `session_id` values on `/api/agent/message` no longer drift
  into phantom in-memory sessions and storage-time foreign-key `500`s; the
  request path now proves the session exists or fails cleanly as not found
- routed provider/model identity is no longer allowed to drift between
  selection and execution; nested specs such as
  `openrouter/openai/gpt-4o-mini` must preserve the outer execution provider
  instead of being silently reinterpreted later as direct `openai` calls

### Less placeholder churn in agent turns

The loop now suppresses more placeholder scaffolding at the execution boundary.

Current improvements:

- `[assistant message]` and `[agent message]` style placeholder outputs are dropped before they can contaminate session history, guard comparison, retries, or user-visible output

This is a framework hygiene fix, not a cosmetic one. Placeholder scaffolding should never be treated as a real completion.

### Less false repetition churn after successful tool work

Cross-turn repetition checks are being tightened to preserve temporal atomicity.

Current improvements:

- guard context now exposes only prior-turn assistant history to
  cross-turn repetition guards
- current-turn tool results remain visible to truth/execution guards without
  poisoning repetition checks
- successful tool-backed confirmations are less likely to get misclassified as
  “repeat of the previous assistant message” simply because the current turn
  has already appended its own assistant content

This is aimed directly at the bad “tool succeeded, then the guard forced
another inference anyway” failure class.

### More disciplined release proof work

The remaining MCP SSE blocker is no longer vague:

- the release now carries a named-target SSE evidence template
- active third-party prospects are explicitly tracked instead of implied
- v1.0.7 closes the remaining SSE release item by narrowing the claim: the
  runtime harness and transport seam are shipped, but proven cross-vendor
  third-party SSE interoperability is not asserted for this release

The behavior-hardening program is also getting a more regimented attack corpus:

- non-idempotent action turns
- multi-artifact authoring
- capability-truth edge cases
- tool-boundary normalization edge cases
- procedural uncertainty / applied learning
- exploratory churn

That keeps the release focused on repeated, demonstrable framework failures
instead of diffuse “agent improvement” talk.

The final hardening pass is also tightening the remaining operator-trust seams:

- output-shape directives such as `return only the number` or `reply on one
  line` are now being separated from semantic subgoals so the verifier stops
  manufacturing fake coverage failures from formatting instructions
- continuity turns are being tightened so durable session history counts as
  first-class evidence when the user asks what they already told the agent in
  the current session
- absolute-path authorization is being corrected to compare canonical path
  identity rather than raw string case, so allowed paths do not fail on
  `/Users/...` vs `/users/...` drift
- direct cron/scheduling requests are being moved onto a focused scheduling
  envelope instead of a general exploratory tool surface
- explicit acknowledgement directives are being tightened onto the shortcut seam
  so one-sentence acknowledgement requests do not expand into broader
  conversational replies
- formatting-only directives are now normalized once and blocked from becoming
  durable unresolved executive questions, so setup turns like `reply only with
  noted` or `reply with only the number 1` do not poison later continuity
  verification
- direct filesystem inspection turns now pin a focused inspection surface
  (`glob_files`, `list_directory`, `read_file`, runtime context) instead of
  depending on semantically lucky default pruning to surface filesystem tools
- inspection-shaped questions now reuse that same focused-inspection detector,
  so vault/workspace follow-up questions do not silently widen into generic
  `question` handling with a broad tool surface and retrieval-default posture;
  path-shaped summary/list/project prompts now ride the same focused
  inspection path instead of being left to generic conversational routing
- focused inspection turns now resolve explicit paths and obvious allowlisted
  aliases such as a single Desktop vault through one shared target-resolution
  seam before inference, so the agent inspects the resolved target directly
  instead of claiming inability when runtime policy already permits access
- explicit path-clarification follow-ups such as `the vault in question is at
  /path/...` now stay on that same focused-inspection seam once the operator
  has supplied the concrete path
- alias-driven inventory prompts such as `what are the most recently updated
  projects in my code folder` now resolve onto the same focused inspection path
  instead of widening into generic question/retrieval behavior
- verifier artifact-overclaim checks now distinguish authored outputs from
  read-only inspection reporting, so listing discovered files in a directory or
  vault no longer degrades the turn as if those files had been claimed as newly
  created artifacts
- authoring turns now resolve allowlisted non-workspace destinations such as
  the Desktop Obsidian vault through the same shared filesystem-target seam as
  inspection, and write-tool metadata now tells the truth about absolute
  allowlisted writes instead of implying workspace-only authoring
- inspection-backed report/inventory authoring now classifies as bounded task
  execution instead of generic creative work, so turns like “inspect my code
  directory, build a report, and write it to my Desktop vault” stay on a
  focused analysis+authoring surface rather than widening to the default
  15-tool path
- the central tool-semantics map now explicitly classifies `search_files` as an
  inspection tool, so bounded focused profiles no longer silently lose it
  through semantic drift
- large code-root reporting now has a first-class project-inventory inspection
  capability, so Duncan no longer has to improvise brittle shell loops or
  extension-globbing heuristics just to assemble project-level metadata before
  writing a report
- request compaction now preserves assistant tool-call messages and their
  matching tool-result messages as one atomic exchange, and that same rule now
  applies to the pipeline-owned pre-inference compactor that mutates live
  session history before retries or later inference, so budget pressure can no
  longer corrupt a tool-bearing turn by dropping the required tool reply and
  forcing a fallback chain after the real work already succeeded
- the live agent loop now threads the authoritative database store into
  `ToolContext`, so DB-backed tools like `cron` no longer fail in production
  with `database store not available` while passing only in isolated tool tests
- artifact-writing turns can now complete with a concise confirmation when the
  written artifact is already proven, instead of degrading only because the
  chat reply did not restate every field that belongs inside the report or
  document itself
- chat-level subgoal coverage now follows that same rule on artifact-backed
  authoring turns, so a successful written report/document is not marked
  degraded only because the final confirmation message is concise; that
  exemption is now scoped to artifact-internal requested parts so mixed-output
  turns still have to satisfy any separate in-chat deliverables
- advisory liveness watchdog events no longer poison terminal status on their
  own; they stay visible in RCA without converting an otherwise successful turn
  into `degraded`
- continuity recall turns now let canonical session-history evidence outrank
  generic retrieval gaps, so simple “what did I tell you earlier” turns stop
  degrading into false uncertainty when the answer is already present in the
  current session
- filesystem inspection tool docs now describe the real sandbox contract:
  workspace-relative paths and absolute allowed paths are both valid when they
  fall within `allowed_paths`

### Cleaner separation between orchestrator and subagents

Subagents are being treated more explicitly as workers rather than operator-facing personalities.

Current improvements:

- subagents no longer inherit the full orchestrator personality footprint
- subagents are being instructed to prove work with evidence rather than style
- orchestrators are being instructed to repackage delegated results for operators instead of passing through raw subagent narration
- multi-subagent delegation now creates a bounded runtime workflow artifact instead of living only as a prompt-side orchestration trick

The intended control flow is:

- operators direct orchestrators
- orchestrators direct subagents
- subagents report upward
- orchestrators present results to operators

### Better local model benchmarking

The benchmarking path now preserves baseline history instead of overwriting it. That gives operators and maintainers a clearer picture of which models are actually viable on current hardware.

This is especially important for local-first operation, where installed models are not automatically usable models.

Benchmark validity is also being tightened:

- baseline runs now need machine-state snapshots at start and finish
- prompt-level exercise rows now need host resource snapshots around each prompt
- prompt-level exercise rows now also need provider/model runtime-state
  snapshots so empty responses can be tied to actual model readiness instead of
  guessed after the fact
- live turn RCA now needs the same host resource evidence at inference time
- wildly different benchmark scores are no longer accepted without the host
  context needed to explain them
- exercise/baseline runs can now target one canonical intent slice from the
  shared matrix, so operators can re-prove capabilities like `TOOL_USE`
  without redefining prompts in the CLI or admin route layer

Recorded follow-on, intentionally deferred beyond `v1.0.7`:

- operator-directed / curated model-selection modes should be formalized later,
  but the current release keeps using the existing routing policy seams and RCA
  to identify where model-selection hardening is actually needed
- if subagent composition later needs “pin the best model for this task type,”
  the orchestrator should consult a read-only router recommendation tool rather
  than reimplement routing in prompt space; that expert-system seam is recorded
  for later work, not added ad hoc in `v1.0.7`
- once the new tool-I/O normalization seam is proven under one in-process
  authority, we may want a mechanism for adding qualified transformers without
  code changes; that extensibility is explicitly deferred until the
  normalization contract, safety boundaries, and RCA visibility are stable

### MCP SSE parity now has a real harness

The remaining MCP SSE parity work is no longer just checklist prose.

v1.0.7 now has:

- one shared MCP config-to-runtime conversion seam
- SSE transport support for endpoint-discovery events and auth-bearing GET/POST
- a named-target `validate-sse` harness that returns structured evidence for a
  configured server

This materially improves RCA quality and operator trust around MCP validation,
but it is not a blanket readiness claim. v1.0.7 explicitly stops short of
claiming proven cross-vendor third-party SSE interoperability because more than
one real external target did not validate through the
same harness.

### A final answer on prompt compression

Prompt compression now has an explicit v1.0.7 disposition instead of lingering
as a soft deferral:

- it stays disabled by default
- it is not recommended for live operator-facing use on the current runtime
- it is retained only for controlled benchmark/comparison work

That is not a punt. The corrected history-bearing soak was decisively negative,
so the right operator-facing answer is to stop presenting prompt compression as
an optimization win when it is not one on this runtime.

### Faster memory retrieval without changing the retrieval contract

The retrieval pipeline is now being tightened so routed memory tiers can run in
parallel while still merging their evidence in deterministic router order.

That matters because the operator should get lower retrieval latency without the
system becoming harder to reason about or explain after the fact.

Retrieval fusion is now explicit:

- routed tier retrieval can stay parallel
- fusion is now the owned stage that combines provenance, freshness,
  authority, route weight, and corroboration
- reranking stays responsible for narrowing and collapse protection instead of
  quietly absorbing more hidden fusion logic

LLM reranking is now explicit too:

- it runs, when enabled, as a bounded semantic stage after fusion
- it uses the same shared LLM service instead of a side path
- it emits dedicated RCA telemetry so operators can see whether it actually ran
- deterministic reranking still remains the fallback and final narrowing path

Semantic retrieval cleanup is now explicit as well:

- semantic retrieval no longer depends on a residual `LIKE` rescue query
- the semantic FTS corpus now indexes category + key + value, preserving
  key-driven semantic lookup on the clean path
- the semantic tier still emits retrieval-path diagnostics, but `LIKE`
  fallback is no longer part of the semantic contract

### More faithful continuity context on resumed turns

Checkpoint restore on the live request path is now closer to the Rust baseline:

- restored checkpoint context now leads with the memory summary instead of a
  digest-dominant note
- active tasks and conversation digest are still preserved
- resumed turns now see a fuller continuity artifact before inference starts

### More explicit learning/consolidation boundaries

The learning side now has a clearer contract too:

- Rust's consolidation baseline is narrower than the current Go distillation path
- Go's broader structured promotion surface is now being kept deliberately,
  not by accident
- repeated high-quality learnings, fix patterns, evidence references, and
  canonical relations can be promoted
- that breadth is bounded by support thresholds, quality gating,
  canonical-vocabulary enforcement, and idempotent writes

### More explicit model policy controls

The runtime is moving away from implicit “hope the router avoids bad models” behavior and toward explicit model lifecycle policy.

Current improvements:

- model policy is operator-visible and becoming tuneable through a persistent policy surface
- routing can now distinguish between models that are generally enabled and models that should only be kept for niche or benchmark-only use
- model restrictions can carry reasons and evidence instead of acting like unexplained hidden policy
- benchmark/exercise selection now uses the same lifecycle policy seam, so models disabled for live or benchmark use are blocked centrally instead of by ad hoc per-command logic
- the WebUI now has the architecture hooks to show lifecycle state and reasons inside existing routing and turn-flow views instead of forcing operators back into logs
- routing evidence is now moving toward recommendation-grade explanations, including callouts when a model was ignored because it lacks capability evidence for the current task class rather than because it was outright blocked
- capability-evidence accounting is being tightened onto one canonical model identity so exercised models are not mislabeled as “unexercised for TOOL_USE” just because benchmark, routing, and provider namespaces drifted apart; that now includes alias resolution between bare routed names, direct provider-qualified names, and nested execution-provider specs
- observability is being tightened so operators can copy full turn ids directly from the trace table, and repeated routing passes are explained causally instead of treating every `routing ×2` as retry churn

### A clearer parity closure program

Parity work for this release is now being treated as an explicit roadmap instead
of a loose collection of old audit notes.

Current improvements:

- the parity ledger is being used as the authoritative system index again
- every remaining parity gap is expected to have its own roadmap entry
- old audit logs are being superseded instead of silently driving scope by
  inertia
- closure is increasingly tied to concrete proof instead of “seems done”

## Current Findings From The First Local Benchmark Cohort

The first fresh local benchmark cohort on the 32 GB M4 test system was terminated early after sufficient negative evidence from `ollama/qwen2.5:32b`.

Current evidence:

- `ollama/gemma3:12b` looks weak and likely unsuitable for live routing on this hardware
- `ollama/gemma4` looks capable but too slow for most interactive orchestrator use
- `ollama/phi4-mini:latest` currently looks strongest in the first local cohort on quality/latency balance, though still slower than ideal for many interactive uses
- `ollama/qwen2.5:32b` looks like a strong retirement candidate for this machine and was dragging both benchmark throughput and live-path health
- `ollama/qwen3:30b-a3b` completed its single-model run but underperformed badly enough to be moved out of live routing contention on this machine and kept only for explicit benchmark/evaluation use

These findings are strong enough to drive current model-policy decisions, but they are not the end of local benchmarking. A second overnight cohort using newly pulled candidate models is planned.
