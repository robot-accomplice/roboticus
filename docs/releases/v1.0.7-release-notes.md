# v1.0.7 Release Notes

Status: draft / in progress

## What This Release Is For

v1.0.7 is focused on three operator-facing outcomes:

1. clearer incident diagnosis when the agent behaves badly
2. more reliable behavior from the orchestrator and subagents
3. better routing decisions, especially on constrained local hardware

This is not a cosmetic release. It is intended to make the system easier to trust, easier to debug, and less likely to waste time on avoidable failures.

## What Operators Should Notice

### Better visibility when a turn goes wrong

The runtime now records much better turn-level diagnostics. When a turn is slow, degraded, or wrong, the system is increasingly able to say:

- what kind of task it thought it was handling
- how much context and tool surface it assembled
- which model/provider path it chose
- whether it was still waiting on the first attempt or stuck in retries
- what fallback or retry behavior occurred

This work is aimed at making failures explainable instead of opaque.

That explanation trail now also has to include the machine state the turn ran
under. Host CPU, memory, swap, and relevant process RSS are being promoted
into the same benchmark and RCA artifacts so operators can tell the difference
between a weak model and a saturated machine.

Operators can now see that explanation trail in the WebUI flow itself instead of reconstructing it from logs:

- the flow view can read the canonical turn diagnostics artifact
- a default macro view now presents a desktop-first left-to-right decision flow grouped into task, envelope, routing, execution, recovery, and outcome
- a visible detail toggle can expose the chronological diagnostic event timeline when deeper RCA is needed
- the old stage flow and RCA split are being collapsed into one bounded decision flow instead of parallel surfaces
- macro blocks stay compact, dense turn state is moving into one narrow status banner instead of a row of large tiles, and turn conclusion plus health can move to a separate bottom banner when space is tight
- macro nodes are being stripped down to one useful visible signal each, with timing as the default and routing as the exception because the chosen model matters more than repeated internal stage names
- verbose text is moving into a true floating tooltip layer plus explicit detail mode
- the flow surface is expected to stay inside the real usable main-pane width, accounting for the sidebar and other persistent chrome, or expose explicit scrolling when it cannot
- if a turn predates canonical diagnostics, the flow now needs to say that explicitly and fall back to a trace-only narrative instead of silently dropping operators back into an unlabeled stage dump
- pipeline traces and canonical diagnostics now need to share the same live turn identity, so fresh turns with RCA data do not get mislabeled as trace-only fallback in the operator UI
- repeated execution is now being promoted into the flow itself with explicit repeat markers on the affected steps
- retry causality is being surfaced directly in the UI: which guard or verifier fired, the retry reason, whether the same model/provider was retried, whether fallback was used, and the final outcome
- the conclusion banner is being upgraded from a telemetry acknowledgment to a real interpretation of what the evidence implies
- the persisted turn-diagnostics summary itself is also being upgraded so downstream UI and RCA consumers are not forced to reinterpret placeholder boilerplate
- flow blocks are gaining outcome coloring from the same RCA evidence: green for clean, yellow for concern, red for broken
- the dense RCA top banner is being aligned to those same semantics as well, including thresholded latency / pressure coloring and an explicitly aggregate health reading rather than a disconnected summary judgment
- the top banner metrics are also gaining explanatory hover/focus tooltips so values like `degraded`, `high`, or host-pressure readings are interpretable without leaving the flow
- stale trace-only fallback overlays are being torn down when the active session or expanded turn changes so old fallback content does not visually contaminate fresh canonical-diagnostics turns

That explanation path is also getting deeper inside verification itself:

- contradiction handling is moving from a boolean hint to a structured evidence artifact
- claim audits are starting to show what proof each high-risk claim required
- missing-proof diagnostics are becoming explicit enough to explain why the system revised or blocked a response
- verifier traces are being promoted from raw claim maps toward operator-legible summary counters for contested claims and proof gaps

### Better behavior on lightweight conversational turns

Simple conversational turns are being handled more cleanly:

- lighter prompt envelopes
- fewer unnecessary tools
- less irrelevant operational/status leakage
- less pointless retry churn

This means short social or conversational prompts should feel less heavy-handed and less error-prone than before.

The same control-plane tightening now applies to simple direct tasks:

- `simple` + `execute_directly` action turns are no longer widened into heavy autonomous tool-bearing turns by the `task` label alone
- retrieval is no longer enabled for imperative turns by default; action turns now need explicit continuity/context cues before memory gets pulled in
- the first pass for simple direct work now stays on a focused execution envelope with a bounded tool surface instead of a heavy autonomous loop

This is specifically aimed at the bad “simple task became a 10-attempt heavy turn” failure class.

This hardening is also closing a more systemic drift class:

- note/document authoring turns no longer get upcast into coding intent just because a filename or heading contains lexical noise like `test`
- DB-enabled skill files are now loaded into the live runtime from their authoritative source paths even when a configured skills directory is wrong or incomplete
- capability fit is now allowed to recognize hyphenated and description-backed skill concepts like `obsidian-vault` instead of treating `obsidian` and `vault` as missing just because there is no exact single-token name match
- when Obsidian is enabled but `vault_path` is blank, the runtime now auto-detects a real workspace-local vault instead of leaving the product in a half-enabled lie

### Explicit Obsidian vault authoring

Obsidian support is no longer expected to work by prompt hint and generic file-tool inference alone.

Current improvements:

- when Obsidian is configured, the live runtime now registers an explicit `obsidian_write` tool
- system prompt guidance now points the model at that named vault-authoring capability
- vault note creation/update now has explicit runtime semantics instead of relying on the model to infer that a generic `write_file` call is acceptable
- workspace-local vaults can now be auto-detected when Obsidian is enabled but the path was left blank, so the runtime can converge on the same truth the operator sees in the workspace layout

This makes vault authoring much more honest as a product capability and reduces the gap between what the operator sees in skills/config and what the runtime can actually do.

Current hardening in this seam is broader than one new tool:

- DB-enabled skills, runtime skill loading, capability fit, prompt guidance,
  and tool registration are being forced to agree on one runtime truth instead
  of drifting across separate inventories
- Obsidian-related capability claims are being tightened so the UI, prompt,
  and live agent do not disagree about whether vault authoring is actually live
- simple note/document authoring turns are being protected from accidental
  upcasts into heavy coding envelopes caused by lexical noise in filenames or
  titles

### Less placeholder churn in agent turns

The loop now suppresses more placeholder scaffolding at the execution boundary.

Current improvements:

- `[assistant message]` and `[agent message]` style placeholder outputs are dropped before they can contaminate session history, guard comparison, retries, or user-visible output

This is a framework hygiene fix, not a cosmetic one. Placeholder scaffolding should never be treated as a real completion.

### Less false repetition churn after successful tool work

Cross-turn repetition checks are being tightened to preserve temporal atomicity.

Current improvements:

- guard context now exposes only prior-turn assistant history to
  cross-turn repetition guards
- current-turn tool results remain visible to truth/execution guards without
  poisoning repetition checks
- successful tool-backed confirmations are less likely to get misclassified as
  “repeat of the previous assistant message” simply because the current turn
  has already appended its own assistant content

This is aimed directly at the bad “tool succeeded, then the guard forced
another inference anyway” failure class.

### Cleaner separation between orchestrator and subagents

Subagents are being treated more explicitly as workers rather than operator-facing personalities.

Current improvements:

- subagents no longer inherit the full orchestrator personality footprint
- subagents are being instructed to prove work with evidence rather than style
- orchestrators are being instructed to repackage delegated results for operators instead of passing through raw subagent narration
- multi-subagent delegation now creates a bounded runtime workflow artifact instead of living only as a prompt-side orchestration trick

The intended control flow is:

- operators direct orchestrators
- orchestrators direct subagents
- subagents report upward
- orchestrators present results to operators

### Better local model benchmarking

The benchmarking path now preserves baseline history instead of overwriting it. That gives operators and maintainers a clearer picture of which models are actually viable on current hardware.

This is especially important for local-first operation, where installed models are not automatically usable models.

Benchmark validity is also being tightened:

- baseline runs now need machine-state snapshots at start and finish
- prompt-level exercise rows now need host resource snapshots around each prompt
- live turn RCA now needs the same host resource evidence at inference time
- wildly different benchmark scores are no longer accepted without the host
  context needed to explain them

### MCP SSE parity now has a real harness

The remaining MCP SSE parity work is no longer just checklist prose.

v1.0.7 now has:

- one shared MCP config-to-runtime conversion seam
- SSE transport support for endpoint-discovery events and auth-bearing GET/POST
- a named-target `validate-sse` harness that returns structured evidence for a
  configured server

This materially improves RCA quality and operator trust around MCP validation,
but it is not yet a blanket readiness claim. Cross-vendor SSE proof remains
open until more than one real third-party target has been validated through the
same harness.

### A final answer on prompt compression

Prompt compression now has an explicit v1.0.7 disposition instead of lingering
as a soft deferral:

- it stays disabled by default
- it is not recommended for live operator-facing use on the current runtime
- it is retained only for controlled benchmark/comparison work

That is not a punt. The corrected history-bearing soak was decisively negative,
so the right operator-facing answer is to stop presenting prompt compression as
an optimization win when it is not one on this runtime.

### Faster memory retrieval without changing the retrieval contract

The retrieval pipeline is now being tightened so routed memory tiers can run in
parallel while still merging their evidence in deterministic router order.

That matters because the operator should get lower retrieval latency without the
system becoming harder to reason about or explain after the fact.

Retrieval fusion is now explicit:

- routed tier retrieval can stay parallel
- fusion is now the owned stage that combines provenance, freshness,
  authority, route weight, and corroboration
- reranking stays responsible for narrowing and collapse protection instead of
  quietly absorbing more hidden fusion logic

LLM reranking is now explicit too:

- it runs, when enabled, as a bounded semantic stage after fusion
- it uses the same shared LLM service instead of a side path
- it emits dedicated RCA telemetry so operators can see whether it actually ran
- deterministic reranking still remains the fallback and final narrowing path

Semantic retrieval cleanup is now explicit as well:

- semantic retrieval no longer depends on a residual `LIKE` rescue query
- the semantic FTS corpus now indexes category + key + value, preserving
  key-driven semantic lookup on the clean path
- the semantic tier still emits retrieval-path diagnostics, but `LIKE`
  fallback is no longer part of the semantic contract

### More faithful continuity context on resumed turns

Checkpoint restore on the live request path is now closer to the Rust baseline:

- restored checkpoint context now leads with the memory summary instead of a
  digest-dominant note
- active tasks and conversation digest are still preserved
- resumed turns now see a fuller continuity artifact before inference starts

### More explicit learning/consolidation boundaries

The learning side now has a clearer contract too:

- Rust's consolidation baseline is narrower than the current Go distillation path
- Go's broader structured promotion surface is now being kept deliberately,
  not by accident
- repeated high-quality learnings, fix patterns, evidence references, and
  canonical relations can be promoted
- that breadth is bounded by support thresholds, quality gating,
  canonical-vocabulary enforcement, and idempotent writes

### More explicit model policy controls

The runtime is moving away from implicit “hope the router avoids bad models” behavior and toward explicit model lifecycle policy.

Current improvements:

- model policy is operator-visible and becoming tuneable through a persistent policy surface
- routing can now distinguish between models that are generally enabled and models that should only be kept for niche or benchmark-only use
- model restrictions can carry reasons and evidence instead of acting like unexplained hidden policy
- benchmark/exercise selection now uses the same lifecycle policy seam, so models disabled for live or benchmark use are blocked centrally instead of by ad hoc per-command logic
- the WebUI now has the architecture hooks to show lifecycle state and reasons inside existing routing and turn-flow views instead of forcing operators back into logs

### A clearer parity closure program

Parity work for this release is now being treated as an explicit roadmap instead
of a loose collection of old audit notes.

Current improvements:

- the parity ledger is being used as the authoritative system index again
- every remaining parity gap is expected to have its own roadmap entry
- old audit logs are being superseded instead of silently driving scope by
  inertia
- closure is increasingly tied to concrete proof instead of “seems done”

## Current Findings From The First Local Benchmark Cohort

The first fresh local benchmark cohort on the 32 GB M4 test system was terminated early after sufficient negative evidence from `ollama/qwen2.5:32b`.

Current evidence:

- `ollama/gemma3:12b` looks weak and likely unsuitable for live routing on this hardware
- `ollama/gemma4` looks capable but too slow for most interactive orchestrator use
- `ollama/phi4-mini:latest` currently looks strongest in the first local cohort on quality/latency balance, though still slower than ideal for many interactive uses
- `ollama/qwen2.5:32b` looks like a strong retirement candidate for this machine and was dragging both benchmark throughput and live-path health
- `ollama/qwen3:30b-a3b` completed its single-model run but underperformed badly enough to be moved out of live routing contention on this machine and kept only for explicit benchmark/evaluation use

These findings are strong enough to drive current model-policy decisions, but they are not the end of local benchmarking. A second overnight cohort using newly pulled candidate models is planned.

## What Changed In Practice

| Area | Operator impact |
| --- | --- |
| Diagnostics | Slow or degraded turns now produce a much better explanation trail. |
| Conversational turns | Lightweight prompts are less likely to be overbuilt or dragged into irrelevant operational context. |
| Delegation | Subagent work is being pushed toward evidence-backed reporting instead of personality-driven narration. |
| Routing | Model choice is increasingly based on task type, runtime evidence, and role fit rather than generic fallback order. |
| Model policy | Models can now be governed more explicitly by role fit, lifecycle state, and stated reasons instead of only by metascore or fallback order. |
| Operator UI | Routing decisions and turn flow now surface lifecycle state, rationale, and a desktop-first macro/detail RCA narrative in-place instead of forcing operators back into logs. |
| Agent UI | The roster and editable subagent list now read from the same enriched subagent projection, so operator-facing agent surfaces stop disagreeing about which agents exist. |
| Prompt compression | The operator-facing settings surface now treats prompt compression as benchmark-only instead of advertising it as a live optimization. |
| Benchmarking | Historical benchmark runs are now preserved for comparison instead of being lost. |
| Retrieval | Routed memory tiers can now fan out concurrently while still merging in deterministic order for stable diagnostics and operator-facing explanation. |
| Continuity | Restored checkpoint context on resumed turns now carries a fuller continuity note instead of a digest-first approximation. |

## What Is Still In Progress

v1.0.7 is not finished yet. Work still open for this release includes:

- publishing the public benchmark report for the first local cohort and running the next deliberately chosen local cohort
- completing the v1.0.7 parity roadmap, including the deferred parity families that were brought back into scope

## Why This Matters

The immediate goal is not just “better metrics.” The goal is:

- fewer slow or irrelevant turns
- clearer explanations when the system does fail
- better use of local models where they are actually viable
- less operator time wasted on preventable routing and delegation mistakes
