# v1.0.8 Release Notes

Status: planned

## What This Release Is For

v1.0.8 is a diagnostic-trust release. Agent fitness, RCA, and benchmarking are
the first gate because every later model, provider, memory, and behavior fix
depends on trustworthy evidence.

This release is not allowed to grow opportunistically. Work is limited to the
items below unless verification exposes a regression that blocks one of them.

## Current Provider-Pack Checkpoint

Provider metadata is configuration-owned. Runtime provider precedence is:

1. operator `roboticus.toml`
2. refreshable `providers_file` / `~/.roboticus/providers.toml`
3. embedded bundled provider defaults

DeepSeek remains OpenAI-compatible provider metadata. Current DeepSeek model
names should be maintained through provider-pack metadata where possible:
`deepseek-v4-flash`, `deepseek-v4-pro`, with `deepseek-chat` and
`deepseek-reasoner` deprecated on 2026-07-24.

## Current Benchmark Checkpoint

Benchmark scorecards separate model efficacy from validity incidents. Transport
and provider-timeout rows remain persisted RCA evidence, but they are excluded
from efficacy scorecards. Legacy blank-class transport rows are normalized
during rescore, and legacy blank/zero/no-class failures are treated as
`validity_ambiguous` rather than resurrected as all-zero model efficacy.

Evaluable benchmark rows now use a default pass floor of `0.50`. Rows below
that threshold are quality-gate failures, not passes, while their measured
quality still remains part of aggregate efficacy evidence. User-configurable
tolerances can adjust policy later, but the default cannot call sub-`.50`
responses successful.

Coding prompt scoring now weights required artifacts ahead of prose shape. The
canonical reverse-string artifact prompt rewards parseable/type-checkable Go
code with actual reversal mechanics and now runs bounded temporary input/output
harnesses for deterministic Go `string -> string`, Python `string -> string`,
and Python in-place list-reversal variants. Type-checkable/non-reversing and
executable-but-incorrect artifacts are penalized below passing quality. Broader
universal compile/run/input-output execution remains the
required direction, but each evaluator must be contract-bound rather than
hidden inside prompt-shape heuristics.

## ABC-Derived Guardrail Items

The ABC paper, "Agent Behavioral Contracts: Formal Specification and Runtime
Enforcement for Reliable Autonomous AI Agents"
([arXiv:2602.22302](https://arxiv.org/pdf/2602.22302)), contains concrete
mechanisms that are better than the current guardrail shape, provided we use
them diagnostically rather than as a broad new DSL. These are promoted into the
v1.0.8 implementation queue one at a time:

1. Contract-shaped guard and RCA events. Current guard output is too often
   reduced to `violations`, `retry`, and `reason`. v1.0.8 should add an
   internal contract event shape that records `contract_id`, `contract_group`,
   phase, precondition state, violation state, recovery action, recovery
   attempt, recovery outcome, and confidence effect. This is the first
   required ABC extraction because it improves evidence quality without first
   changing model behavior.

2. Hard/soft invariant classification. Every guard or verifier rule should be
   classified as hard, soft, or neutral. Hard invariants block or return
   structured failure; soft invariants recover within a bounded window or
   degrade with evidence; neutral conditions do not retry, punish, or persist
   false learning. This directly protects the operating principle that useful
   imperfect answers are better than unnecessary no-answer outcomes.

3. Recovery-window metrics. Retry count is not sufficient evidence. RCA and
   benchmark rows should record which contract failed, what recovery path was
   selected, whether recovery completed within the allowed window, and whether
   useful work was preserved. This separates model capability from framework
   recovery behavior.

4. Contract drift metrics for soaks. Long-running sessions should track repeated
   soft violations, hard violations by intent class, recovery success rate,
   recovery attempts per successful turn, unsupported-certainty trend, false
   capability-denial trend, and memory-contradiction trend. Aggregate pass/fail
   is not enough to diagnose behavioral degradation.

5. Memory behavior contracts. Memory retrieval and learning should be governed
   by explicit contracts: retrieve only when continuity or non-derivable facts
   are needed, treat absent memory as neutral, keep derivable or mutable facts
   out of durable confidence, and route contradictions to uncertainty plus
   targeted correction rather than destructive cleanup.

6. R-TEOR-R contract composition. Contract events should attach to Retrieve,
   Think, Execute, Observe, Reflect, and Remember phases so failures can be
   attributed to the phase that caused them. Public ROVER terminology remains
   acceptable, but implementation evidence must not collapse the two Remember
   phases or hide phase-specific defects.

ABC changes require paired soak evidence before they can be treated as
release-complete. The canonical soak plan is
[ABC Contract Soak Plan](../testing/abc-contract-soak.md). The baseline is
`b20bb69b`; the after lane must run the same scenario list, cache mode, server
mode, model configuration, and latency ceilings. Behavior reports alone are not
enough: the isolated RCA database must also be retained so contract severity,
recovery, phase, and confidence-effect metrics can be compared.
Any gain or loss attributed to ABC must come from a comparison marked
`valid_for_abc_attribution: true`; otherwise the result is diagnostic evidence
only, not an ABC causality claim.

## Agent Behavior Corrections

### Reflect / continuation working memory

- **Trailing system overlay**: TOTOF and continuation briefs are built with
  `ContextBuilder.BuildRequestWithTrailingSystemOverlay` so the full
  conversation (`session.Messages()`) stays in the LLM request and the brief is
  appended as the last system message(s). This replaces the old
  `BuildRequestWithMessages` scaffold that dropped prior user, assistant, and
  tool turns during reflect and continuation.
- **`CONTINUE_EXECUTION` handling**: Sentinel detection is line-anchored (the
  token may appear at the start of any line after leading prose). Prose before
  the sentinel and the control line itself are not persisted as the
  user-visible assistant answer. `scrubControlSentinels` is applied on
  assistant-message persistence paths as a second line of defense.

- False generic capability denials are now treated as trust defects, not
  acceptable fallback prose. If a turn has successful runtime/tool evidence, the
  assistant may only claim missing access or missing tools when a concrete
  policy, sandbox, network, provider, or tool error proves that limitation.
  Live prompt guidance now describes that rule without embedding the exact
  denial phrase as model-facing prose. Bounded guard retries now fail closed on
  unresolved hard trust defects instead of returning a known-false final answer.
- Resolved inspection tasks may not finalize with zero inspection evidence.
  When the operator gives a concrete repo/path target and the selected turn
  surface includes filesystem/read tools, the assistant must inspect, report a
  concrete tool/policy/sandbox denial, or retry through the normal guard path;
  it may not ask the operator to initiate the scan on its behalf.
- Complex repo inspection and architecture/code-review requests stay on direct
  execution instead of being upcast into delegation by generic skill-token
  mismatch or routed into the generic heavy/default envelope merely because
  they are complex. Delegation repair remains tracked separately; it must not
  weaken the orchestrator's own authoritative inspection surface or remove the
  focused filesystem/read tools needed to finish the requested review.
- Magic-phrase reduction is now an explicit agent-fitness rule: exact phrases
  may be weak detector hints or protocol/safety evidence, but continuity and
  guard behavior must be justified by structured turn state. The first
  implementation target is broad-to-narrow follow-up continuity where the
  operator refers to prior assistant next steps without using a blessed
  confirmation phrase.
- Promissory execution scaffolding is not a final answer. Responses ending in
  “let me test/check…” after only introspection or runtime-context evidence are
  retried because they neither execute nor report an observed result.
- Inspection-backed answers must carry concrete observed entries into the final
  response when the operator asked for a listing or summary. A reflection that
  says “a clear list was observed” while omitting the list is classified as an
  output-contract defect.
- Recovered guard failures remain diagnosable. If a hard guard catches and
  repairs a response, RCA may still mark the turn degraded, but the summary now
  names the recovered guard violation instead of leaving `primary_diagnosis`
  blank.

## Memory Checkpoint

- Ternary memory confidence is covered at the typed verification-evidence
  boundary: contradictions remain confidence-lowering, absent memory remains
  neutral, and actual retrieved evidence can reinforce. The verifier also keeps
  derivable in-turn answers neutral instead of treating missing memory as a
  proof failure.
- Memory Curation is now the umbrella term for post-turn memory lifecycle work.
  The legacy `consolidate` command/API path remains for compatibility, but
  mutating curation work is skipped when sessions are active inside the
  quiescence window.
- Existing consolidation coverage continues to mark derivable tool-output
  memories stale so mutable/current facts degrade out of active evidence rather
  than becoming durable false support.

## CLI Topology Checkpoint

- Root help is now grouped into operator-facing domains: Runtime & Operations,
  Configuration & Security, Models, Agents & Memory, Automation & Channels,
  Administration & Maintenance, and Help.
- Existing top-level command paths remain callable for backward compatibility.
  v1.0.8 does not move command paths; any future path migration must ship
  aliases or proxy commands for every moved command.

## Dead-Code Checkpoint

- Planned-but-unused helper wrappers and dormant subsystem methods are treated
  as architecture defects. The release removes the unused root CLI table
  wrapper and the old dedicated Telegram poller because table rendering already
  has a live owner in `cmd/internal/cmdutil`, and Telegram polling is owned by
  the channel listener through `router.PollAll()`.

## Release Gate

The release cannot be considered feature-complete until:

- benchmark/RCA validity defects are closed or explicitly excluded from model
  efficacy scoring
- coding benchmark scoring has functional correctness coverage for at least the
  active canonical coding prompt set where executable checks are feasible
- canned-response and overconstraint regressions have tests
- memory-confidence semantics are protected by tests
- provider-pack precedence is tested
- at least one known-good Ollama model and one previously failing or
  unavailable Ollama model are exercised after classification fixes, proving
  both efficacy scoring and validity-only exclusion
- `roboticus upgrade all` and `roboticus mechanic --repair` both prove
  idempotent cleanup of stale updater state, stale sidecars, and known schema
  drift, with concrete action summaries rather than silent mutation
- release notes, architecture docs, regression matrix, and site changelog are
  updated

## Live Exercise Findings To Shore Up

During the targeted `moonshot/kimi-k2.6` v1.0.8 exercise run, passing rows are
still being treated as diagnostic input rather than assumed success. Findings
to investigate before closing the benchmark/RCA gate:

1. `TOOL_USE:C2` originally passed with low quality after responding with an
   intended database lookup rather than an observed count. This is now closed
   at the benchmark-scoring seam: required-tool prompt contracts distinguish
   observed or blocked evidence from promissory "I will query/check/search"
   text.

2. `DELEGATION:C3` exposed a second scoring-contract defect: action prompts
   could score highly even when the answer said the task could not be completed,
   offered copy-paste instructions instead of performing the action, or reported
   that the task failed. This is now guarded in scorer tests as diagnostic
   evidence rather than clean completion.

3. `TOOL_USE:C3` exposed another required-tool scoring defect: an access or
   allowlist failure that prevents the requested file discovery was treated as
   successful tool-use evidence. This is now closed for scoring: legitimate
   negative lookup results can pass, but configuration/access/tool failures
   that block the requested work cannot pass.

4. `EXECUTION:C4` exposed the same action-completion scoring class at expert
   complexity: policy-denied filesystem access plus a reference blueprint was
   scored as a clean execution pass. Policy/tool-denial prose is now capped
   below the pass floor unless concrete completion evidence exists.

5. `DELEGATION:C4` passed despite `Q=0.42`. That exposed a pass-threshold
   defect: evaluable rows below `.50` must fail by default while preserving
   their quality score for RCA and aggregate efficacy.

Benchmark scoping correction:

- Exact prompt-row replay and explicit warm-up control are part of the
  v1.0.8 benchmark/RCA gate. Scope diagnosis should target one inference such
  as `TOOL_USE:C2` without running all five prompts in an intent slice, and
  local model diagnosis must let operators choose whether to include cold/warm
  transition cost.

Agent behavior correction:

- Runtime filesystem context now explicitly treats configured allowed paths as
  inherited roots, and the guard layer blocks false claims that a child path
  must be separately allowlisted unless tool evidence shows an actual sandbox
  denial.
- Focused inspection turns must preserve the resolved project/root path for
  the whole turn. Relative follow-up calls such as `docs/`, `internal/`, or
  `ARCHITECTURE.md` are anchored to the resolved inspection root instead of
  falling back to the generic workspace.
- Sessions > Context now treats request-footprint trace metadata as first-class
  context evidence. Missing legacy snapshot rows are displayed as coverage
  gaps, not as proof that no context diagnostics exist.
- Sessions > Context turn detail now treats the request footprint as an
  inspectable allocation artifact: the visualization is a vertical context
  budget bar with always-visible slice labels and a right-side selected-slice
  detail pane, so operators can see both the total pressure and the payload
  behind the selected system/tools/memory/history/user/reflection slice.
- Tool-call audit persistence is idempotent for replayed execution records so
  live RCA cannot lose tool evidence to duplicate-key warnings.

## Active Research Extraction Queue

- Recursive Language Models / RLM paper investigation
  ([arXiv:2512.24601v1](https://arxiv.org/html/2512.24601v1)) already captured
  for v1.0.8. Implementation requires a near-term measurable behavior target
  and a plan to avoid reinforcing bad memories.
- Agent Behavioral Contracts / ABC paper
  ([arXiv:2602.22302](https://arxiv.org/pdf/2602.22302)) is promoted into the
  active guard/agent-fitness workstream. The evaluation target is not a new
  feature surface; it is to extract anything useful for improving or replacing
  current guardrail methodology. Candidate mechanisms include:
  - preconditions, hard/soft invariants, governance policies, and recovery
    mechanisms as explicit runtime contracts
  - probabilistic satisfaction semantics (`p`, tolerance `delta`, recovery
    window `k`) for measuring imperfect-but-recoverable behavior
  - drift/compliance/recovery metrics that can feed RCA, benchmark fitness, and
    memory-learning confidence
  - contract composition rules for orchestrator/subagent and multi-step
    R-TEOR-R flows

  Incorporation is allowed only if it yields a concrete contract model,
  measurable reliability improvement, and no regression to canned fallback
  behavior.
