# v1.0.0 Release Footprint — "The Go Rewrite"

**Date**: 2026-04-11
**Previous release**: v0.11.4 (Rust)
**Thesis**: Complete ground-up rewrite in Go with full Rust parity + new capabilities. This is the cutover release.

---

## Versioning Rationale

v0.11.4 was the final Rust release. The Go rewrite is not incremental — it's a new codebase (140K LoC, 687 Go files, 329 test files, 3232 test functions) that replaces the Rust implementation entirely. Semantic versioning says: **v1.0.0** signals the stable public API is established.

If v1.0.0 feels premature, **v0.20.0** is an alternative that signals "major leap, not yet 1.0."

## Parity Policy

**Exact Rust replication is the release gate.** Every gap is fixed to match Rust behavior exactly.

1. **No improvisation** — port the Rust implementation faithfully before attempting improvements
2. **No "Go is better" divergence** — if the Go code does something differently, it's a bug until proven otherwise
3. **~48 "Go-unique" items must be re-evaluated** — items previously marked "document as Go improvement" (stricter injection markers, broader shell validation, configurable loop thresholds, etc.) must be audited: do they diverge from Rust? If yes, align to Rust first
4. **If Rust behavior seems wrong** — flag it, implement it anyway, fix it in a post-parity enhancement pass with evidence
5. **Thresholds, algorithms, defaults, formats, error messages** — everything matches exactly

### Track 3: In-Chat `/status` Overhaul (MUST — parity shortfall)

Current `/status` response:
```
Status: Agent Duncan is online. Session: abc123, Messages: 5
```

This is a serious regression from the Rust codebase. The in-chat `/status` must surface the same operational data as the CLI `status` command:

| Section | Data |
|---------|------|
| **Agent** | Name, status, current model, uptime |
| **Providers** | Count configured, each with state (healthy/degraded/down) |
| **Channels** | Each platform with connected/disconnected status |
| **Sessions** | Active count, current session turn count |
| **Memory** | Tier counts (working/episodic/semantic/procedural), retrieval health |
| **Cron** | Job count, failed count, next fire time |
| **Cache** | Hit rate, entry count |
| **Wallet** | Balance, token, chain |
| **Skills** | Enabled/total count |

Format: Compact but rich. Use channel-appropriate formatting (Telegram markdown, Discord embeds, plain text fallback). The CLI already has all the API calls — port the data gathering to `cmdStatus()` in `bot_commands.go`.
