## Summary

Two distinct streams converged into v1.0.6.

**Memory architecture closure.** Milestones 3 (semantic FTS+vector read path) and 8 (relational distillation from enriched episodes) reached acceptance. The agentic memory roadmap is now materially complete except for the operator-driven LIKE retirement gate (M3.3 telemetry surface shipped; deletion deferred until production traces confirm the safety net is dormant).

**Operational hardening.** A v1.0.5 incident (`sudo roboticus daemon stop` failed with opaque launchctl errors, then chowned the user's data dir to root, then a fresh restart hit a broken SQLite migration, then revealed an ambient-rogue-DB bug, then surfaced a Playwright MCP `initialize failed: EOF` with no diagnostic context) cascaded across six unrelated subsystems. v1.0.6 fixes every one of them and lands the diagnostic surfaces (system-warnings dashboard banner, MCP child-stderr capture, pipeline stage liveness watchdog) that turn future incidents from cascades into single-line operator-actionable signals.

Guiding principle for the operational stream: **silent failures aren't really failures from the system's perspective — operators are the only ones who can act, and they need loud signal to act on.**

## Phase 2 — Operational Hardening

### DB layer

| Item | Description |
|------|-------------|
| Migration 042 SQLite-compliant | The original `ALTER TABLE ... ADD COLUMN updated_at TEXT NOT NULL DEFAULT (datetime('now'))` violated SQLite's "no non-constant default on ADD COLUMN" rule and failed every fresh install with `Cannot add a column with non-constant default (1)`. Rewritten using the canonical SQLite pattern (ADD nullable + UPDATE backfill); existing installs at schema_version ≥ 42 are skipped by the runner. |
| File permissions to 0o600 | `db.Open()` now chmods the database file plus WAL sidecars (`-wal`, `-shm`) to 0o600 after schema init. Pre-v1.0.6 files inherited the process umask (0o644 on macOS — world-readable). Best-effort: chmod failures (e.g., file owned by another user from a prior sudo invocation) log a warning but don't block boot. R-DBPERMS-1..4. |

### Daemon control

| Item | Description |
|------|-------------|
| PID-file based stop | `roboticus daemon stop` reads `~/.roboticus/roboticus.pid` (or `cfg.Daemon.PIDFile` if set), sends SIGTERM, waits up to 30s for graceful shutdown, escalates to SIGKILL if needed. No sudo. No DB open. Idempotent — stop on a not-running daemon returns nil. Recovers automatically from `kill -9` zombie state. |
| Minimal `NewServiceOnly` constructor | Install / Uninstall / Control / Status no longer call `daemon.New()` (which ran the entire 12-step boot sequence just to construct a service handle). Eliminates the chown-side-effect chain that turned a single sudo invocation into a multi-day recovery story. |
| macOS uses `launchctl bootout` | Bypasses kardianos/service's legacy `launchctl unload` path (which returns "Input/output error" for stale services). Iterates system/ → gui/<uid>/ domains so installed-as-system and installed-per-user paths both resolve. |
| R-DAEMON-PID-1..3, R-DAEMON-STOP-1..4, R-DAEMON-LAUNCHCTL-1, R-DAEMON-NOBOOT-1 | Lock the contract end-to-end. |

### Operator visibility

| Item | Description |
|------|-------------|
| System-warnings collector | Process-singleton at `core.AddSystemWarning` / `SystemWarningsSnapshot`. Dedupes on (Code, Detail) so warnings recorded by multiple subsystems collapse to one operator-visible entry. Concurrent-safe via internal mutex; snapshots are independent copies (safe for HTTP polling). R-WARNINGS-1..6. |
| Boot-time warning prompt | New `bootSystemWarningsBanner` step in the 12-step boot output. Renders any collected warnings just before "Ready" so foreground operators see them immediately. |
| Dashboard banner endpoint | `GET /api/admin/system-warnings` returns `{warnings: [], count: 0}` (never null) for the dashboard's TypeScript types to consume. R-WARNINGS-7..8. |
| Triggers | `WarningCodeConfigDefaultsUsed` (severity HIGH) when `initConfig` falls through to defaults — surfaces the silent-rogue-DB failure mode that produced the v1.0.5 audit findings. `WarningCodeDatabaseCreatedAtPath` (severity HIGH) when `db.Open` lazily materializes a new file at an unexpected path. R-WARNINGS-9..10. |

### MCP

| Item | Description |
|------|-------------|
| Stdio diagnostic surface | `NewStdioTransport` now captures child stderr into a bounded ring-style buffer (most recent 8KB). A second goroutine reaps the child via `cmd.Wait()` so exit code is observable. `StdioTransport.ChildDiagnostic()` composes both into a human-readable summary that `ConnectStdio`'s failure paths fold into the returned error. Pre-v1.0.6 a child crash produced only `mcp: initialize failed: EOF`; post-v1.0.6 the same failure surfaces npm's actual stderr (e.g., `404 Not Found`, `Cannot find module`). R-MCP-DIAG-1..3. |
| Pipeline stage liveness watchdog | `Pipeline.Run` spawns a goroutine that polls `TraceRecorder.CurrentSpan` every 10s. Stages running longer than 20s produce a `WARN pipeline stage running longer than expected` log every probe interval thereafter. Operators can identify stuck stages from live daemon output instead of needing `kill -QUIT` and a goroutine dump. R-PIPE-WATCHDOG-1..4. |
| Agent system-prompt directives | Two new directives close the v1.0.5 `filesystem_count_only` behavioral regression: (a) ATTEMPT then report — agent must call the tool and let the policy engine return the actual decision rather than reasoning preemptively about constraints; (b) honor explicit output-format requests verbatim ("only the number" → deliver that shape, no preface). R-AGENT-PROMPT-1..2. |

## Known Open Items (post-v1.0.6)

These were deliberately deferred to keep the release scoped and grounded:

- **M3.3 LIKE-block deletion**: telemetry surface (`AggregateRetrievalPaths`) is in place; the actual deletion in `retrieval_tiers.go` waits for production traces to demonstrate dormancy per the documented retirement procedure.
- **External behavioral soak rerun pending**: the three pass-5 residuals (`introspection_discovery`, `tilde_distribution`, `filesystem_count_only`) are now addressed in code via an introspection alias, explicit `~` refusal in both tool resolution and policy evaluation, and stripping parsed tool-call JSON from assistant content. Release confidence should be refreshed with another managed clone/fresh soak pass before declaring the behavioral matrix fully closed.
- **Fresh-state soak lane**: the managed `fresh` lane now boots from an isolated generated default config rather than a copied operator config, so clean-state validation is finally meaningful. Long-duration fresh-state soak evidence is still an rc validation artifact rather than something pinned in this notes file.
- **Prompt compression remains suspect until paired-soak evidence exists**: the repo now includes `scripts/run-prompt-compression-soak.py`, which runs the same live scenarios with prompt compression forced `off` and `on` against isolated configs and treats pass→fail drift as a regression. That harness is the closure bar for prompt compression; wiring the feature is not sufficient evidence of quality.
- **Prompt compression is intentionally narrower than the old Rust-era surface**: the current Go implementation compresses only older conversational history and leaves system prompt, memory, ambient system notes, and tool payload messages verbatim. This is deliberate protection for the richer Go memory architecture, not an accidental divergence.
- **Cross-vendor SSE MCP target**: validated only against in-tree fixture this release. If a production third-party SSE MCP server becomes available before v1.0.7, the checklist's item 3 should be re-run with that target named.

## Upgrade Notes

- **Existing installs**: the DB file permission tightening (`db.Open` → 0o600) auto-applies on next boot. No operator action required.
- **Existing installs with rooted files**: if `~/.roboticus/state.db` or any file in the data dir is owned by `root` (from a pre-v1.0.6 `sudo roboticus daemon stop`), the chmod will warn-but-not-block. Run `sudo chown -R "$(id -u):$(id -g)" ~/.roboticus` to restore ownership; future invocations will not need sudo.
- **`roboticus upgrade all` behavior change**: providers.toml and skills/*.md are now PRESERVED by default. Pass `--refresh-config` to opt into the previous overwrite behavior. Behavior change is documented in the command's `--help` output.
- **`roboticus daemon stop` no longer needs sudo** when stopping a daemon you started yourself (`roboticus serve` or via launchd in user-mode). System-mode launchd installs still need sudo for the underlying `launchctl bootout`.
- **MCP config**: if your `roboticus.toml` references `@anthropic/mcp-server-playwright`, replace with `@playwright/mcp` — the original package name returns 404 from npm.
